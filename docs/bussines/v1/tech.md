# Технологии, стек и технические ограничения

## Формат продукта
- Тип продукта: только веб-приложение.
- Текущий scope: web-first.
- Мобильные приложения вне scope MVP.
- Адаптивность допустима, но основной фокус — удобство desktop web.

## Frontend
- Фреймворк: Next.js.
- Язык: TypeScript.
- Стратегия рендеринга: Next.js App Router с использованием серверных возможностей там, где это уместно; приложение не проектируется как чистая SPA по умолчанию.
- Стилизация: Tailwind CSS.
- UI-компоненты: shadcn/ui.
- Загрузка данных и управление server state: TanStack Query.
- Тяжелый глобальный state manager на старте не использовать без реальной необходимости.
- Предпочтение: простой локальный state + server state.

## Backend
- Язык backend: Go.
- Архитектурный стиль: модульный монолит.
- HTTP router/framework: Chi.
- Стиль API: REST JSON API.
- Backend должен быть отдельным сервисом относительно frontend.

## База данных
- Основная база данных: PostgreSQL.
- Доступ к базе: pgx.
- ORM не использовать.
- Работа с запросами: явные SQL-запросы / query builder с полным контролем над запросами.
- Миграции обязательны.
- Инструмент миграций: golang-migrate.
- На уровне данных должна поддерживаться история изменений.

## Аутентификация
- Метод аутентификации на текущем этапе: email/password.
- Система должна изначально поддерживать мультипользовательскую архитектуру.
- OAuth-провайдеры вне scope данной итерации.

## Инфраструктура и деплой
- Модель деплоя: self-hosted.
- Контейнеризация: Docker.
- Локальная и dev-оркестрация: Docker Compose.
- Система должна проектироваться так, чтобы работать без зависимости от cloud-only сервисов.

## Тестирование и качество
- Тесты backend обязательны.
- Обязательные типы тестов на backend:
  - unit tests
  - integration tests
- Архитектурный приоритет: чистота архитектуры и поддерживаемость важнее скорости реализации.

## Архитектурные приоритеты
- Приоритет: чистая структура, явные границы модулей, долгосрочная поддерживаемость.
- Предпочтение: модульный монолит вместо микросервисов.
- Не усложнять инфраструктуру в MVP без необходимости.
- Не добавлять преждевременно аналитику, вложения и сложные коллаборативные функции в рамках данной итерации.

## Что не входит в эту итерацию
- Нет мобильного приложения.
- Нет вложений и файлов.
- Нет детальной бизнес-логики.
- Нет сложной аналитики.
- Нет сложной клиентской state-архитектуры без подтвержденной необходимости.

# Технические нефункциональные требования

## Общие требования
- Система должна быть спроектирована как поддерживаемое и расширяемое веб-приложение.
- Приоритет: чистота архитектуры, предсказуемость поведения, простота сопровождения.
- MVP не должен содержать избыточной инфраструктурной или архитектурной сложности.
- Все решения должны быть совместимы с дальнейшим развитием продукта без полной переработки ядра.

## Архитектура
- Архитектурный стиль backend: модульный монолит.
- Backend и frontend должны быть развернуты как отдельные сервисы.
- Границы доменных модулей должны быть явными.
- Бизнес-логика не должна смешиваться с транспортным слоем, ORM-логикой или UI-логикой.
- Код должен быть организован так, чтобы доменные правила можно было тестировать изолированно.

## API
- Внешний интерфейс backend: REST JSON API.
- API должно быть консистентным по структуре маршрутов, кодам ответа и формату ошибок.
- Все входные данные должны валидироваться на сервере.
- Ошибки валидации должны возвращаться в предсказуемом и структурированном виде.
- API не должно полагаться на клиент как на источник истины для бизнес-ограничений.
- Сервер обязан самостоятельно проверять доменные инварианты.

## Работа с данными
- Основная база данных: PostgreSQL.
- Доступ к данным должен быть реализован без ORM.
- SQL-запросы должны быть явными, контролируемыми и понятными.
- Все изменения схемы БД должны проходить через миграции.
- Миграции должны быть воспроизводимыми и применимыми в автоматическом режиме.
- Структура БД должна поддерживать мультипользовательскую изоляцию данных.
- История изменений должна сохраняться отдельно от основной предметной модели.
- Физическое удаление допустимо для основных сущностей, но не должно ломать согласованность данных.

## Консистентность и целостность
- Система должна гарантировать корректность доменных ограничений на уровне backend.
- Необходимо предотвращать создание циклов в DAG-зависимостях тем.
- Нельзя допускать нарушение ссылочной целостности между сущностями.
- Все операции изменения связанных сущностей должны выполняться предсказуемо и безопасно.
- Там, где это необходимо, должны использоваться транзакции.

## Производительность
- MVP не требует сложной оптимизации под высокую нагрузку.
- Тем не менее, система не должна проектироваться заведомо неэффективно.
- Базовые операции пользователя должны выполняться без заметных задержек при нормальном объеме личных данных.
- Запросы к спискам тем, задач и материалов должны быть реализованы так, чтобы не создавать избыточных N+1 сценариев.
- Вычисление прогресса и сводок должно быть реализовано достаточно эффективно для обычного персонального использования.

## Безопасность
- Система должна поддерживать аутентификацию по email/password.
- Пароли должны храниться только в безопасном хешированном виде.
- Доступ к данным пользователя должен быть строго изолирован.
- Пользователь не должен иметь возможности получить доступ к данным другого пользователя ни через API, ни через косвенные идентификаторы.
- Все защищенные маршруты должны требовать аутентификации.
- Сервер не должен доверять user_id, переданному с клиента без проверки контекста аутентификации.

## Telegram-уведомления
- Telegram-бот должен быть отдельным исполняемым компонентом рядом с backend либо отдельным модулем в рамках backend-инфраструктуры.
- Уведомления должны формироваться автоматически по расписанию один раз в день утром.
- Сбой отправки уведомления не должен приводить к падению всего приложения.
- Логика уведомлений должна быть отделена от основной доменной логики.
- Механизм планирования уведомлений должен быть пригоден для self-hosted среды.

## Тестирование
- Unit-тесты backend обязательны.
- Integration-тесты backend обязательны.
- Бизнес-правила должны покрываться тестами отдельно от HTTP-слоя.
- Интеграционные тесты должны проверять работу с PostgreSQL и ключевые сценарии изменения данных.
- Критические доменные сценарии должны быть покрыты тестами до уровня уверенного рефакторинга.
- Для MVP e2e-тесты не являются обязательными, если отдельно не согласованы.

## Логирование и наблюдаемость
- Backend должен иметь базовое структурированное логирование.
- Ошибки приложения должны логироваться в понятном и пригодном для отладки виде.
- Критические фоновые операции, включая отправку уведомлений, должны логироваться отдельно.
- Логи не должны содержать пароли, токены и другие чувствительные данные.
- Для MVP достаточно базовой наблюдаемости без сложного стека мониторинга.

## Конфигурация
- Конфигурация приложения должна передаваться через environment variables.
- Все чувствительные значения должны храниться вне исходного кода.
- Конфигурация должна поддерживать как локальную разработку, так и self-hosted deployment.
- Должен существовать понятный шаблон конфигурации для запуска проекта.

## Docker и запуск
- Проект должен запускаться в self-hosted окружении через Docker.
- Для локальной разработки и базового деплоя должен использоваться Docker Compose.
- Должен существовать воспроизводимый способ поднятия:
  - frontend
  - backend
  - PostgreSQL
  - фонового процесса/бота уведомлений
- Новый разработчик должен иметь возможность поднять проект локально с минимальным числом ручных шагов.

## Качество кода
- Кодовая база должна быть единообразной по стилю.
- Должны использоваться линтеры и базовые инструменты форматирования.
- Следует избегать скрытой магии, чрезмерной абстракции и сложных паттернов без необходимости.
- Имена сущностей, методов и модулей должны прямо отражать бизнес-смысл.
- Архитектурные компромиссы должны быть локальными и явными.

## Расширяемость
- Архитектура должна позволять в будущем добавить:
  - дополнительные типы уведомлений,
  - более сложную работу с материалами,
  - новые представления roadmap,
  - дополнительные способы аутентификации.
- При этом MVP не должен заранее включать абстракции, не подтвержденные текущими требованиями.
- Проектировать нужно с запасом на рост, но без преждевременного усложнения.
