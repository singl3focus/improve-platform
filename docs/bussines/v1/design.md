Нужно спроектировать **веб-платформу для личного обучения и развития по roadmap**.
Это не корпоративный таск-менеджер и не универсальный Notion-клон.
Продукт должен ощущаться как **персональный центр обучения**, где пользователь:

* двигается по своей roadmap,
* видит зависимости между темами,
* отслеживает прогресс,
* работает с задачами,
* хранит учебные материалы,
* получает ощущение ясности, структуры и движения вперед.

### Основная идея интерфейса

Интерфейс должен быть:

* минималистичным,
* современным,
* спокойным,
* сфокусированным,
* без визуального шума,
* без ощущения “корпоративной CRM”,
* без перегруженных таблиц и сложных панелей.

Продукт должен восприниматься как **инструмент для глубокой личной работы**, а не как “офисный софт”.

### Визуальный стиль

Стиль: **clean, modern, calm, premium productivity**.

Ощущение:

* структурность,
* ясность,
* контроль,
* прогресс,
* концентрация,
* мягкая мотивация без агрессивной геймификации.

Избегать:

* кислотных цветов,
* чрезмерной иллюстративности,
* слишком “игрового” UI,
* ощущения enterprise/B2B dashboard,
* перегруженных sidebar + toolbar + widgets layouts.

### Цветовая палитра

Предпочтительно:

* светлая тема как основная,
* нейтральная палитра,
* много воздуха,
* мягкие серые,
* белый или почти белый фон,
* один акцентный цвет для состояний прогресса и активных элементов.

Акцентный цвет:

* спокойный синий, сине-фиолетовый или изумрудный.

Статусы и состояния:

* не начато — нейтральный серый,
* в работе — акцентный цвет,
* на паузе — мягкий янтарный,
* завершено — спокойный зеленый,
* просрочено — приглушенный красный без агрессии.

### Типографика

* современная sans-serif,
* хорошая читаемость,
* крупные и ясные заголовки,
* аккуратная визуальная иерархия,
* комфортные отступы,
* не использовать слишком плотный интерфейс.

### Общий UX-подход

Платформа должна быть построена вокруг **трех центральных сущностей**:

1. **Roadmap**
2. **Темы и их прогресс**
3. **Задачи и материалы**

Главная цель UX:
пользователь должен за 2–3 секунды понимать:

* где он сейчас,
* что изучает,
* что заблокировано зависимостями,
* что нужно сделать сегодня,
* какой у него прогресс.

## Основные экраны

### 1. Главный экран / Dashboard

Главный экран должен сразу показывать личный контекст пользователя.

Что должно быть на экране:

* текущий общий прогресс по roadmap,
* темы “в работе”,
* ближайшие задачи с дедлайнами,
* заблокированные темы (из-за зависимостей),
* материалы, к которым пользователь недавно возвращался,
* краткая сводка дня.

Композиция:

* чистый header,
* компактный sidebar,
* основная область из карточек/блоков,
* без перегруженности,
* приоритет на главные действия.

Dashboard должен выглядеть как **персональная рабочая панель обучения**, а не как аналитический BI-экран.

### 2. Экран Roadmap

Это ключевой экран продукта.

Roadmap должна визуализироваться как **гибкая карта тем с зависимостями**, а не просто как линейный список.

Важно:

* логика roadmap — DAG,
* темы объединены в этапы,
* этапы — это только визуальные группы,
* зависимости между темами явно видны,
* интерфейс должен помогать понимать путь прохождения.

Возможные визуальные принципы:

* канвас или graph-like layout,
* группировка тем по этапам,
* карточки тем внутри этапов,
* стрелки/связи между карточками,
* хорошая читаемость зависимостей,
* не перегружать визуализацию.

Карточка темы должна показывать:

* название,
* статус,
* прогресс,
* краткую метаинформацию,
* признак блокировки,
* количество задач,
* количество материалов.

Важно сохранить баланс:

* roadmap должна выглядеть “умно” и структурно,
* но не быть слишком сложной, технической или похожей на mind map editor.

### 3. Экран темы

Экран темы — это рабочее пространство конкретной темы.

Что должно быть видно:

* название темы,
* статус,
* прогресс,
* описание,
* зависимости,
* checklist / задачи по теме,
* материалы по теме,
* ключевые даты.

Структура экрана:

* верхняя часть: основная информация о теме,
* ниже: блок задач/checklist,
* рядом или ниже: материалы,
* отдельный блок зависимостей.

Экран должен ощущаться как **страница учебного модуля**, а не просто карточка задачи.

### 4. Экран задач

Нужен отдельный экран задач в формате **kanban**.

Колонки:

* новая
* в работе
* на паузе
* выполнена

Просроченность:

* не отдельная колонка,
* а визуальный индикатор на карточке.

Карточка задачи:

* название,
* связанная тема (если есть),
* дедлайн,
* индикатор просрочки,
* краткое описание.

Kanban должен быть:

* чистым,
* простым,
* не перегруженным,
* более “личным”, чем “командным”.

### 5. Экран материалов

Материалы — важная часть платформы, не второстепенная.

Это библиотека учебных ресурсов:

* книги,
* курсы,
* видео,
* статьи,
* любые другие источники.

Но в MVP не нужно жестко типизировать материалы через UI.
Лучше показать их как **гибкие карточки ресурсов**.

Что отображать:

* название,
* описание,
* связанная тема,
* прогресс по материалу,
* порядок/позиция внутри темы.

Экран должен ощущаться как **личная curated library**, а не как файловый менеджер.

## Компоненты и паттерны

### Карточки тем

Карточки должны быть аккуратными, с:

* мягкими тенями или тонкими границами,
* скруглениями,
* ясными статусами,
* визуальным акцентом на прогресс.

### Прогресс

Прогресс должен быть заметным, но не кричащим:

* progress bars,
* небольшие кольцевые индикаторы,
* проценты только там, где действительно нужны.

### Зависимости

Зависимости — одна из ключевых особенностей продукта.
Они должны быть визуально понятны:

* стрелки,
* линии,
* бейджи “заблокировано”,
* счетчик prerequisite topics.

### Sidebar

Sidebar должен быть компактным и спокойным.

Разделы:

* Dashboard
* Roadmap
* Темы
* Задачи
* Материалы
* Настройки

Без лишних пунктов.

### Header

Header минимальный:

* название текущего раздела,
* глобальный поиск или quick add,
* профиль пользователя.

## Поведение интерфейса

Интерфейс должен поддерживать:

* быстрое добавление темы,
* быстрое добавление задачи,
* быстрое добавление материала,
* простое изменение статуса,
* понятное редактирование связей между темами,
* drag-and-drop там, где он действительно полезен.

Важно не делать интерфейс “тяжелым редактором”.
Пользователь должен чувствовать, что всё легко обновляется и поддерживается в актуальном состоянии.

## Эмоциональное ощущение продукта

Пользователь должен чувствовать:

* “я понимаю, куда иду”,
* “я вижу свой путь”,
* “я не перегружен”,
* “у меня есть структура”,
* “я двигаюсь вперед”.

Продукт должен уменьшать хаос в обучении и давать ощущение упорядоченного развития.
