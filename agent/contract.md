# Agent Contract (единый источник правил)

## 0) Назначение
Этот документ задаёт обязательные правила работы кодовых агентов в данном репозитории:
- как читать и изменять код,
- как фиксировать задачи и прогресс,
- как описывать изменения,
- как фиксировать проверенные выводы,
- какие запреты и инварианты нельзя нарушать.

Если требования в других местах противоречат этому документу — действует **agent/contract.md**.

---

## 1) Мышление: всё — гипотеза до проверки
Любое утверждение о поведении системы/кода считается гипотезой, пока не подтверждено:
- тестом,
- запуском команды,
- статическим анализом,
- чтением точного места в коде (путь+строка/функция),
- логами/трассировкой,
- воспроизводимым сценарием.

### 1.1. Минимальный цикл работы
1) Сформулировать гипотезу (что и почему не так / что улучшить).
2) Получить доказательство (как минимум один проверяемый сигнал).
3) Внести минимальные изменения.
4) Повторно проверить сигнал (что стало лучше и не сломалось).
5) Документировать результат.

---

## 2) Жёсткие запреты (Hard Constraints)
### 2.1. Запрет на git
Агент **НЕ ИСПОЛЬЗУЕТ git**:
- не выполняет `git status`, `git diff`, `git commit`, `git push`, `git checkout`, `git merge` и т.п.
- не создаёт ветки, не коммитит, не пушит.

✅ Вместо этого агент:
- делает изменения в файловой системе репозитория (редакция файлов),
- формирует **предложение сообщения для коммита** (commit message suggestion) в формате, принятом проектом.

### 2.2. Запрет на непроверенные выводы
- Нельзя добавлять в отчёты/задачи утверждения вида “скорее всего”, “наверное”, “кажется”, если нет проверки.
- Нельзя утверждать, что тесты/линтеры пройдены, если команда не запускалась (или нет подтверждения).

### 2.3. Минимизация поверхностных изменений
- Не делайте “косметические” рефакторы, если они не нужны для задачи.
- Не смешивайте несколько смыслов в одной задаче.
- Дифы должны быть минимальными для достижения критериев готовности.

---

## 3) Источники правды в репозитории (must-read)
Перед началом любой работы агент должен прочитать:
- `agent/contract.md` (этот файл)
- `agent/task.md` (шаблон задачи)
- `agent/changes_description.md` (правила описания изменений)

---

## 4) Задачи: единый формат и статусы
Задачи хранятся в:
- `agent/tasks/frontend/`
- `agent/tasks/backend/`

### 4.1. Допустимые статусы
- `Todo`
- `In Progress`
- `Review Ready`
- `Finished`
- `ReMake`

### 4.2. Обязательные поля задачи
Каждая задача должна иметь:
- `ID` (например `TSK-0001`)
- `Status`
- `Type` (`Business` или `Technical`)
- `Title`
- `Context` (что и почему)
- `Acceptance Criteria` (проверяемые критерии)
- `Constraints` (важные ограничения/запреты)
- `Verification` (как проверить: команды/тест-кейсы/сигналы)
- `Change Summary` (итоговое описание по шаблону)
- `Suggested Commit Message` (одно сообщение для коммита)
- `Notes` (Manager/Frontend-Developer/Backend-Developer/Reviewer)

### 4.3. Правило обновления задач
- Менеджер **добавляет** задачи (и может уточнять формулировки), но не переводит их в “готово”.
- Frontend-Developer меняет статус задач в `agent/tasks/frontend/`: `Todo/ReMake` → `In Progress` → `Review Ready`.
- Backend-Developer меняет статус задач в `agent/tasks/backend/`: `Todo/ReMake` → `In Progress` → `Review Ready`.
- Ревьюер меняет `Review Ready` → `Finished` или `ReMake`.

---

## 5) Definition of Done (DoD)
Задача считается корректно выполненной только если:
1) Выполнены `Acceptance Criteria`.
2) Указана проверка в `Verification` и её результат (или честно указано, что команда не запускалась).
3) Изменения описаны по правилам из `agent/changes_description.md`.

---

## 6) Риски и безопасность
Агент должен явно отмечать:
- изменения, влияющие на безопасность (auth, secrets, ACL),
- изменения, влияющие на данные (миграции, форматы),
- изменения, влияющие на производительность и контракты API.

Если есть риск — добавьте в задачу “Rollback plan” (как быстро откатить) и “Risk notes”.

---

## 7) Стиль коммуникации
- Если данных недостаточно для решения — агент задаёт конкретные вопросы в задаче (в разделе Notes → Questions).
- Не “угадывать” требования: сначала уточнить или зафиксировать предположения как гипотезы и проверить.

---

## 8) Если правила конфликтуют
- Применяйте более строгие требования.
- Если конфликт блокирует работу — зафиксировать в Notes и задать вопрос владельцу репозитория/постановщику задачи.
