### Change Summary

**Problem**
- В backend-сервисе отсутствовал механизм уведомлений о дедлайнах задач. Для MVP требуется ежедневная утренняя сводка, отправляемая через Telegram-бота, с информацией о просроченных и сегодняшних задачах.

**Root Cause Hypothesis**
- Не применимо (greenfield-функциональность; таблица `tasks` с полем `deadline` уже создана в миграции TSK-0002).

**Evidence**
- До начала работы в `internal/` не было модуля `notify/`, конфигурация не содержала Telegram-параметров, `main.go` не запускал фоновый процесс уведомлений.

**Solution**
Создан модуль `internal/notify/` (5 файлов + 2 файла тестов) и интегрирован в `cmd/server/main.go`:

Структура `internal/notify/`:
- `model.go` — доменные модели `DeadlineTask`, `DailySummary`, интерфейсы `TaskQuerier` (1 метод) и `Sender` (1 метод).
- `repo.go` — `Repo` на `*pgxpool.Pool`: `GetDeadlineTasks(ctx, today)` — запрос задач с `deadline IS NOT NULL AND status != 'done' AND deadline <= today`, разделение на overdue (deadline < today) и due today (deadline == today), сортировка `ORDER BY deadline, created_at`.
- `telegram.go` — `TelegramSender`: обёртка над `github.com/go-telegram/bot v1.19.0`. `NewTelegramSender(token, chatID, log)` создаёт бота. `SendDailySummary(ctx, summary)` форматирует сводку через `FormatDailySummary` и отправляет в Telegram-чат с HTML-разметкой.
- `formatter.go` — `FormatDailySummary(summary)` формирует HTML-сообщение: заголовок с датой, секция Overdue (с количеством дней просрочки), секция Due Today. `escapeHTML` защищает от XSS в заголовках задач. Возвращает пустую строку если нет задач (сообщение не отправляется).
- `scheduler.go` — `Scheduler`: фоновый процесс с time-based scheduling. `Run(ctx)` вычисляет время следующего запуска (`nextRun`), ожидает через `time.Timer`, выполняет `execute`, повторяет. `execute` запрашивает задачи через `TaskQuerier`, формирует `DailySummary`, отправляет через `Sender`. Ошибки запроса и отправки логируются через `slog.Error`, но не прерывают работу планировщика. `ExecuteNow(ctx)` — публичный метод для немедленного запуска (тестирование/ручной триггер). Планировщик останавливается по отмене контекста.

Scheduler:
- Запускается из `cmd/server/main.go` как goroutine перед `srv.Run()`.
- Работает только если `TELEGRAM_BOT_TOKEN` задан (если пуст — логируется и пропускается).
- Останавливается при завершении основного сервиса (через `context.WithCancel` + `defer cancel`).

Конфигурация:
- `TELEGRAM_BOT_TOKEN` — токен Telegram-бота (опциональный; если пуст, уведомления отключены).
- `TELEGRAM_CHAT_ID` — ID чата для отправки (обязателен если TOKEN задан; валидируется в config).
- `NOTIFY_HOUR` — час отправки (default 9).
- `NOTIFY_MINUTE` — минута отправки (default 0).
- `NOTIFY_TIMEZONE` — часовой пояс расписания (default "UTC"). При невалидном значении fallback к UTC с логированием ошибки.

**Risks & Rollback**
- Риск: scheduler запрашивает ВСЕ задачи с дедлайнами `<= today` по всем пользователям. Для MVP с одним пользователем (self-hosted) это приемлемо. Для production с множеством пользователей потребуется per-user Telegram chat ID (отдельная таблица + миграция) и итерация по пользователям.
- Риск: scheduler делает дополнительный SELECT один раз в день — влияние на производительность минимально.
- Риск: при недоступности Telegram API сообщение будет потеряно (нет retry/queue). Для MVP приемлемо; для production рекомендуется добавить retry с exponential backoff или очередь сообщений.
- Риск: `NOTIFY_TIMEZONE` с невалидным значением fallback к UTC с логированием, но не блокирует запуск.
- Откат: удалить `internal/notify/`, откатить изменения в `config.go`, `config_test.go`, `main.go`, `.env.example`, удалить `github.com/go-telegram/bot` из `go.mod`.

**Config changes**
- Новые опциональные env: `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID` (int64), `NOTIFY_HOUR` (int, default 9), `NOTIFY_MINUTE` (int, default 0), `NOTIFY_TIMEZONE` (string, default "UTC").
