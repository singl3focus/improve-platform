## TSK-0012 — Автоприменение миграций отдельным контейнером в Docker Compose
- Status: Finished
- Type: Technical
- Approval: Approved
- Created: 2026-03-15
- Updated: 2026-03-15 (Reviewer)

### Context
- Сейчас миграции применяются вручную после старта стека (через `psql` из `build/README.md`), а не как отдельный сервис в `docker-compose`.
- Требование owner: при `docker compose up -d --build` миграции должны подниматься каждый раз отдельным контейнером и применяться к БД.
- Для целостности среды backend должен стартовать только после успешного завершения миграций.

### Acceptance Criteria
- [x] В `build/docker-compose.yml` добавлен отдельный сервис миграций (`migrations`), который запускается при `docker compose up -d --build` и применяет миграции из `backend/migrations` через `golang-migrate`.
- [x] Повторный запуск `docker compose up -d --build` не падает на уже примененных миграциях (поведение идемпотентно: no-op/успешное завершение).
- [x] `backend` не стартует, если сервис миграций завершился с ошибкой; при успешных миграциях `backend` проходит health-check.
- [x] `build/README.md` и связанные инструкции обновлены: ручной шаг применения миграций удален, описан поток через migration-контейнер.

### Constraints
- Не использовать `git`-операции.
- Не менять бизнес-логику backend API; изменения ограничить инфраструктурой запуска и документацией.
- Сохранить запуск в self-hosted режиме через Docker Compose без cloud-only зависимостей.
- Миграции должны применяться штатным инструментом проекта (`golang-migrate`), без ручного `psql` как основного сценария.

### Change Summary
**Problem**
- Миграции в локальном Docker Compose применялись вручную через `psql` из `build/README.md`, что ломало требуемый сценарий `docker compose up -d --build` без дополнительных шагов.
- `backend` не был явно привязан к успешному завершению отдельного migration-процесса.

**Root Cause Hypothesis**
- В `build/docker-compose.yml` отсутствовал отдельный сервис миграций на `golang-migrate`.
- В `build/README.md` оставался устаревший manual flow для применения SQL.

**Evidence**
- Код:
  - `build/docker-compose.yml:24` — добавлен сервис `migrations` на `migrate/migrate`.
  - `build/docker-compose.yml:54` — `backend` зависит от `migrations` через `condition: service_completed_successfully`.
  - `build/README.md:19` и `build/README.md:35` — удалён ручной шаг `psql`, описан поток через migration-контейнер.
- Команды:
  - `docker compose -f build/docker-compose.yml config` — валидный compose-конфиг с сервисом `migrations`.
  - `docker compose -f build/docker-compose.yml up -d --build` (2 запуска) — без ошибок.
  - `docker compose -f build/docker-compose.yml logs migrations --timestamps --tail=200` — есть `1/u initial_schema ...` и затем `no change`.

**Solution**
- В `build/docker-compose.yml`:
  - добавлен сервис `migrations` (`migrate/migrate:v4.18.3`), который запускает `up` по файлам из `../backend/migrations` после `postgres:service_healthy`;
  - для `backend` добавлена зависимость `migrations: service_completed_successfully`, чтобы API не стартовал до успешных миграций.
- В `build/README.md`:
  - удалена инструкция ручного применения SQL через `psql`;
  - добавлено описание автоприменения миграций контейнером `migrations`;
  - уточнена проверка: `migrations` завершается `Exited (0)`, в логах допустимы `applied`/`no change`.

**Tests / Verification**
- `docker compose -f build/docker-compose.yml config` — PASS.
- `docker compose -f build/docker-compose.yml up -d --build` — PASS.
- Повторный `docker compose -f build/docker-compose.yml up -d --build` — PASS (идемпотентный сценарий миграций).
- `docker compose -f build/docker-compose.yml logs migrations --timestamps --tail=200` — PASS (`1/u initial_schema`, затем `no change`).
- `docker compose -f build/docker-compose.yml ps` — PASS (`postgres` healthy, `backend` healthy/running).
- `curl.exe -sS -i http://localhost:8080/readyz` — PASS (HTTP 200, `{"status":"ok"}`).

**Impact**
- API-контракт: без изменений.
- Данные: миграции теперь применяются автоматически штатным инструментом (`golang-migrate`) в compose-потоке.
- Обратная совместимость: да (изменения ограничены `build/docker-compose.yml` и `build/README.md`).

**Risks & Rollback**
- Риск: если новая миграция падает, `backend` не стартует (ожидаемое поведение по требованиям).
- Rollback:
  - вернуть предыдущую версию `build/docker-compose.yml` (без `migrations` и без зависимости `service_completed_successfully`);
  - вернуть предыдущую инструкцию в `build/README.md` с ручным шагом применения миграций.
