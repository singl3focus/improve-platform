## TSK-0021 — Удалить stages из backend roadmap и контракта API
- Status: Finished
- Type: Business
- Approval: Approved
- Created: 2026-03-17
- Updated: 2026-03-17

### Context
- Owner подтвердил полный отказ от этапов (stages) в roadmap: stages мешают UX и должны быть удалены с backend и frontend.
- В текущем backend stages являются частью схемы БД, доменной модели и API-контрактов roadmap.
- Без удаления stages на backend фронтенд не сможет перейти на чистый граф тем без stage_id.

### Acceptance Criteria
- [x] Из backend удалены stage-сущности и stage-операции roadmap: доменная модель, usecase/repo/handler, роуты `/api/v1/roadmap/stages*`.
- [x] Контракт `GET /api/v1/roadmap` больше не содержит stages-обертку; ответ отдает граф тем и зависимостей в согласованном формате без `stage_id`.
- [x] Операции создания/редактирования тем работают без `stage_id` и сохраняют инварианты roadmap (user scope, DAG, блокировки, статусные правила).
- [x] Обновлена существующая миграция `000001_initial_schema.up.sql` под модель без таблицы `stages` и без FK `topics -> stages` (новые миграции не добавляются).
- [x] Обновлены unit/handler/usecase тесты roadmap-модуля и маршрутизации; `go test ./...` проходит.

### Constraints
- Не добавлять новые миграции; разрешено обновить существующую `000001_initial_schema.up.sql`.
- Сохранить проверки user scope, cycle detection и бизнес-валидации статусов тем.
- Изменение контрактов roadmap является осознанно breaking для BFF/frontend и должно быть явно отражено в задаче frontend.
- Не использовать git-команды.

### Change Summary
**Problem**
- Backend roadmap зависел от stages на всех уровнях: схема БД, модели, API и роутинг, что блокировало переход на stage-free UX.

**Root Cause Hypothesis**
- Исторически roadmap был реализован как вложенная структура `stages -> topics`, поэтому `stage_id` стал обязательной частью topic CRUD и контракта `GET /api/v1/roadmap`.

**Evidence**
- До изменений stage-роуты были зарегистрированы в `backend/internal/server/server.go` (`/api/v1/roadmap/stages*`).
- До изменений stage-сущности и `stage_id` присутствовали в `backend/internal/roadmap/model.go`, `backend/internal/roadmap/usecase.go`, `backend/internal/roadmap/repo.go`, `backend/internal/roadmap/handler.go`.
- Выполнена проверка: `go test ./internal/roadmap ./internal/server` -> PASS.
- Runtime-сценарии подтверждены тестами:
  - `TestHandler_GetRoadmap_ResponseDoesNotContainStageFields` (нет `stages`/`stage_id` в JSON).
  - `TestFrontendRoadmapStageRoutes_Removed` (legacy `/api/v1/roadmap/stages*` возвращают `404`).
  - `TestHandler_CreateTopic_SuccessWithoutStageFields` и `TestHandler_UpdateTopic_SuccessWithoutStageFields` (topic create/update без stage-полей).
  - `TestHandler_UpdateTopicStatus_Blocked` и `TestHandler_AddDependency_CycleRejected` (инварианты блокировки/зависимостей).
- Выполнена регрессионная проверка: `go test ./...` -> PASS.

**Solution**
- Удалены stage-сущности и stage-операции из roadmap domain/service/repository/handler.
- Удалены stage endpoints из роутинга сервера.
- Контракт `RoadmapResponse` переведён на stage-free формат: топ-уровневые `topics` и `dependencies`.
- Topic create/update и create-with-dependency переведены на вход без `stage_id`.
- SQL в roadmap repo переведён на таблицу topics без `stage_id`.
- Обновлена миграция `backend/migrations/000001_initial_schema.up.sql`: удалена таблица `stages`, удалены `topics.stage_id`, FK на `stages` и индекс по `stage_id`.
- Для ReMake добавлены smoke-тесты контракта/маршрутизации: отсутствие `stages`/`stage_id` в JSON roadmap, отсутствие stage endpoints (`404`) и успешные topic create/update без stage-полей.

**Tests / Verification**
- Запущено: `go test ./internal/roadmap ./internal/server`.
- Результат: PASS.
- Запущено: `go test ./...`.
- Результат: PASS.
- API smoke покрыт автоматизированными handler/server тестами (httptest); e2e/BFF smoke отдельно не запускался.

**Impact**
- API: breaking change для roadmap-контракта и topic create/update payload (без `stage_id`).
- Data model: initial schema roadmap теперь без таблицы `stages` и без FK `topics -> stages`.
- Business rules: проверки user scope, DAG cycle detection и status transition сохранены.
- Обратная совместимость: нет (осознанный breaking change).

**Risks & Rollback**
- Риск: несинхронизированный frontend/BFF, который ожидает `stages` и `stage_id`, перестанет корректно работать.
- Rollback plan: вернуть изменения в roadmap-контракте/роутинге и `000001_initial_schema.up.sql` к модели со stages.

**Suggested Commit Message**
- backend: remove roadmap stages and switch to stage-free topic graph contract
