## TSK-0002 — Схема БД и миграции для ядра MVP
- Status: Finished
- Type: Technical
- Approval: Approved

### Context
- Для MVP нужны явные SQL-схемы с поддержкой мультипользовательской изоляции и истории изменений.
- Требование проекта: PostgreSQL + `golang-migrate`, без ORM.

### Acceptance Criteria
- [x] Добавлены миграции `golang-migrate` для таблиц: users, roadmap, stages, topics, topic_dependencies, tasks, materials, history (technical/business events).
- [x] На уровне схемы обеспечена ссылочная целостность и user-level изоляция данных.
- [x] Добавлены индексы для типовых выборок (topics/tasks/materials по `user_id`, дедлайны задач, зависимости тем).
- [x] Миграции воспроизводимо применяются вверх/вниз без ручных правок.

### Constraints
- Физическое удаление основных сущностей допустимо, но не должно ломать согласованность.
- Циклы в зависимостях контролируются логикой backend (не только БД).
- Любые несовместимые изменения схемы требуют явных migration notes в Change Summary.

### Change Summary

**Problem**
- В репозитории отсутствовала схема БД. Для реализации бизнес-логики MVP необходимы таблицы с ссылочной целостностью, user-level изоляцией и историей изменений.

**Root Cause Hypothesis**
- Не применимо (greenfield-задача).

**Evidence**
- До начала работы директория `backend/migrations/` не существовала, SQL-схемы не было.

**Solution**
Создана единая начальная миграция `000001_initial_schema` в `backend/migrations/`:

Таблицы (8):
- `users` — email/password auth, UNIQUE(email).
- `roadmaps` — одна на пользователя, UNIQUE(user_id), UNIQUE(id, user_id) для composite FK, FK → users ON DELETE CASCADE.
- `stages` — UI-группы тем, `user_id` денормализован для composite FK, FK (roadmap_id, user_id) → roadmaps(id, user_id), UNIQUE(id, user_id), position.
- `topics` — ядро обучения, FK → users, composite FK (stage_id, user_id) → stages(id, user_id), UNIQUE(id, user_id), ENUM `topic_status` (not_started, in_progress, paused, completed), даты (start/target/completed), position.
- `topic_dependencies` — связи DAG, composite PK (topic_id, depends_on_topic_id), `user_id` для изоляции, composite FK обоих сторон → topics(id, user_id), CHECK на self-reference.
- `tasks` — единая модель для standalone задач и checklist items, composite FK (topic_id, user_id) → topics(id, user_id) (MATCH SIMPLE — при topic_id IS NULL FK не проверяется), ENUM `task_status` (new, in_progress, paused, done), deadline DATE, position.
- `materials` — ресурсы привязанные к теме, composite FK (topic_id, user_id) → topics(id, user_id), progress 0–100 (CHECK), position.
- `history_events` — техническая и бизнес-история, entity_type + entity_id, event_type CHECK IN ('technical', 'business'), payload JSONB.

Индексы (12 явных + 13 constraint-based = 25 доменных):
Явные:
- `idx_stages_roadmap_id` — этапы по roadmap.
- `idx_topics_user_id`, `idx_topics_stage_id` — выборки тем по пользователю и этапу.
- `idx_topic_deps_depends_on` — обратный поиск зависимостей.
- `idx_tasks_user_id`, `idx_tasks_topic_id`, `idx_tasks_deadline` (partial, WHERE deadline IS NOT NULL) — выборки задач и дедлайнов.
- `idx_materials_user_id`, `idx_materials_topic_id` — выборки материалов.
- `idx_history_user_id`, `idx_history_entity`, `idx_history_created_at` — выборки истории.
Constraint-based (13): 8 pkey + users_email_unique + roadmaps_one_per_user + roadmaps_id_user + stages_id_user + topics_id_user.

Ссылочная целостность и user-level изоляция (composite FK):
- `stages_roadmap_user_fk`: (roadmap_id, user_id) → roadmaps(id, user_id) — stage не может ссылаться на roadmap чужого пользователя.
- `topics_stage_user_fk`: (stage_id, user_id) → stages(id, user_id) — topic не может ссылаться на stage чужого пользователя.
- `tasks_topic_user_fk`: (topic_id, user_id) → topics(id, user_id) — task не может ссылаться на topic чужого пользователя.
- `materials_topic_user_fk`: (topic_id, user_id) → topics(id, user_id) — material не может ссылаться на topic чужого пользователя.
- `topic_deps_topic_fk` + `topic_deps_depends_on_fk`: обе стороны зависимости привязаны к одному user_id.
- ON DELETE CASCADE на всех FK — удаление пользователя каскадно удаляет все его данные.

- 4 негативных SQL-сценария (кросс-пользовательские ссылки): все отклонены с FK constraint error.
- Тестовое окружение: PostgreSQL 16-alpine в Docker, порт 5433.

**Impact**
- Новые файлы: `backend/migrations/000001_initial_schema.up.sql`, `backend/migrations/000001_initial_schema.down.sql`.
- Изменения схемы: полная начальная схема MVP.
- Обратная совместимость: не применимо (первая миграция).

**Risks & Rollback**
- Риск минимальный: greenfield, нет существующих данных.
- Откат: `migrate down 1` полностью удаляет все таблицы и типы.

**Migration Notes**
- Начальная миграция. Несовместимых изменений нет.
