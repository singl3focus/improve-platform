## TSK-0003 — Email/password auth и изоляция данных пользователя
- Status: Finished
- Type: Business
- Approval: Approved
- Created: 2026-03-14
- Updated: 2026-03-14

### Context
- MVP мультипользовательский архитектурно, при этом каждый пользователь работает только со своими данными.
- Без auth невозможно безопасно открыть бизнес-эндпоинты frontend.

### Acceptance Criteria
- [x] Реализованы регистрация и логин по email/password с безопасным хранением хеша пароля.
- [x] Защищенные API-маршруты требуют аутентификацию.
- [x] Любые выборки/модификации данных фильтруются по `user_id` из auth-контекста, а не из клиента.
- [x] Попытка доступа к чужим данным возвращает корректную ошибку авторизации/доступа.

### Constraints
- OAuth вне scope MVP.
- Не логировать пароль, хеш, токены и секреты.
- Сервер не доверяет `user_id` из payload/query/path без проверки контекста сессии.

### Change Summary

**Problem**
- В backend-сервисе отсутствовала аутентификация. Все будущие бизнес-эндпоинты требуют авторизованного доступа и user-level изоляции данных.

**Root Cause Hypothesis**
- Не применимо (greenfield-функциональность поверх каркаса TSK-0001).

**Evidence**
- До начала работы в `internal/` не было модуля `auth/`, конфигурация не содержала `JWT_SECRET`, маршрутизация не включала auth-эндпоинты и protected-группу.

**Solution**
Создан доменный модуль `internal/auth/` (6 файлов + 3 файла тестов):

Структура:
- `model.go` — модель `User`, DTO (`TokenResponse`, `UserResponse`), sentinel errors (`ErrUserNotFound`, `ErrEmailExists`, `ErrInvalidCredentials`), интерфейсы `Repository` и `Service`.
- `repo.go` — `UserRepo` на `*pgxpool.Pool`: `Create` (с обработкой unique_violation → `ErrEmailExists`), `FindByEmail`, `FindByID`.
- `usecase.go` — `UseCase`: `Register` (bcrypt hash + JWT), `Login` (bcrypt verify + JWT), `GetCurrentUser` (по ID из контекста). Токен: HS256, TTL 24h.
- `handler.go` — `Handler` с HTTP-обработчиками: `Register()` (POST, 201), `Login()` (POST, 200), `Me()` (GET, 200, protected). Валидация: email/password обязательны, пароль ≥ 8 символов.
- `middleware.go` — `Middleware(jwtSecret)` Chi-middleware: извлекает JWT из `Authorization: Bearer <token>`, валидирует подпись и срок, записывает `user_id` (claim `sub`) в `context.Context`. Хелперы `UserIDFromContext(ctx)` и `WithUserID(ctx, id)` (для тестов).

Эндпоинты:
- `POST /api/v1/auth/register` — публичный, регистрация, возвращает `{"access_token":"..."}`.
- `POST /api/v1/auth/login` — публичный, логин, возвращает `{"access_token":"..."}`.
- `GET /api/v1/me` — protected (JWT middleware), возвращает `{"id":"...","email":"...","created_at":"..."}`.

Изменённые файлы:
- `internal/config/config.go` — добавлено поле `JWTSecret` (mapstructure `JWT_SECRET`), валидация обязательности.
- `internal/config/config_test.go` — все существующие тесты обновлены для `JWT_SECRET`, добавлен `TestLoad_MissingJWTSecret`.
- `internal/server/server.go` — тип `db DBPinger` заменён на `pool *pgxpool.Pool` (совместимо, `pgxpool.Pool` удовлетворяет `handler.DBPinger`). В `routes()` инициализируется auth-модуль (`repo → usecase → handler`), добавлена protected-группа Chi с `auth.Middleware`.
- `.env.example` — добавлен `JWT_SECRET=change-me-to-a-random-secret`.
- `go.mod` / `go.sum` — добавлены `golang.org/x/crypto v0.49.0`, `github.com/golang-jwt/jwt/v5 v5.3.1`.

Изоляция данных:
- JWT middleware извлекает `user_id` из подписанного токена (claim `sub`) и помещает в `context.Context`.
- `UserIDFromContext(ctx)` — единственный способ получить `user_id` в handler/usecase; значение из клиентского payload/query/path игнорируется.
- На уровне БД: composite FK из TSK-0002 блокируют кросс-пользовательские ссылки.
- Без валидного JWT middleware возвращает `401 {"error":{"code":"unauthorized","message":"..."}}`.

**Impact**
- Новые файлы: `internal/auth/` (model.go, repo.go, usecase.go, handler.go, middleware.go, usecase_test.go, handler_test.go, middleware_test.go).
- Изменённые файлы: `internal/config/config.go`, `internal/config/config_test.go`, `internal/server/server.go`, `.env.example`, `go.mod`, `go.sum`.
- API: три новых endpoint (`/api/v1/auth/register`, `/api/v1/auth/login`, `/api/v1/me`).
- Конфигурация: новый обязательный env `JWT_SECRET`.
- Обратная совместимость: сигнатура `server.New` изменилась (`DBPinger` → `*pgxpool.Pool`); `main.go` уже передавал `*pgxpool.Pool`, изменения вызывающего кода не требуются.

**Risks & Rollback**
- Риск: добавлен обязательный env `JWT_SECRET` — сервис не запустится без него. Убедитесь, что `JWT_SECRET` задан во всех средах.
- Откат: удалить `internal/auth/`, вернуть `server.go`, `config.go`, `config_test.go`, `.env.example` к предыдущей версии, удалить новые зависимости из `go.mod`.

**Config changes**
- Новый обязательный env: `JWT_SECRET` — секрет для подписи JWT-токенов (HS256). Не должен быть пустым.

**Security**
- Пароли хранятся как bcrypt-хеш (DefaultCost=10); открытый пароль в памяти только на время хеширования/сравнения.
- JWT-токен подписан HS256; секрет читается из env, не хардкожен.
- Middleware не логирует значения токенов/секретов.
- Ошибки авторизации не раскрывают, существует ли email (единое сообщение "invalid email or password").
