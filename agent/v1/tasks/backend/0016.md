## TSK-0016 — Разрешить перевод задачи в `done` из любого незавершенного статуса
- Status: Finished
- Type: Business
- Approval: Approved
- Created: 2026-03-15
- Updated: 2026-03-15

### Context
- Сейчас backend ограничивает переходы статусов задачи и может возвращать `invalid task status transition` при попытке закрыть задачу из части состояний.
- По требованию owner задача должна переводиться в "выполнена" из любого рабочего состояния.
- Нужна корректировка доменных правил переходов в `task` модуле с обновлением тестов.

### Acceptance Criteria
- [x] `PATCH /api/v1/tasks/{taskID}/status` со статусом `done` успешен для задач в статусах `new`, `in_progress`, `paused`.
- [x] Ошибка `invalid task status transition` больше не возникает при переводе в `done` из указанных статусов.
- [x] Существующие бизнес-инварианты модуля `task` (включая автообновление прогресса/статуса темы) не регрессируют.
- [x] Обновлены unit/handler тесты для новых допустимых переходов.
- [x] `go test ./...` в `backend/` проходит успешно.

### Constraints
- Не менять API-формат и коды ошибок вне контекста переходов в `done`.
- Не менять схему БД и миграции.
- Изменение должно быть локализовано в `task` домене и его тестах.

### Verification
- Command/Test:
  - `cd backend && go test ./...`
  - Expected: все тесты проходят, включая новые кейсы переходов в `done`.
  - Result: PASS (после запуска вне sandbox: `ok improve-platform/internal/task 0.873s`; общий прогон `./...` без падений).
  - Manual API smoke:
  - Expected: задача переводится в `done` из `new`, `in_progress`, `paused` без `invalid_status`.
  - Result: NOT RUN

### Change Summary
**Problem**
- Переход в `done` был разрешён не из всех незавершённых статусов задачи: доменная валидация могла возвращать `invalid task status transition` для части рабочих сценариев.

**Root Cause Hypothesis**
- В `task` usecase матрица `validTransitions` была слишком строгой: `done` разрешался только из `in_progress`.

**Evidence**
- `backend/internal/task/usecase.go` (map `validTransitions`): до изменения
  - `new -> in_progress`
  - `in_progress -> paused, done`
  - `paused -> in_progress, new`
- `backend/internal/task/usecase_test.go`: тест `TestUpdateTaskStatus_InvalidTransition` фиксировал `new -> done` как невалидный переход.

**Solution**
- В `backend/internal/task/usecase.go` расширены допустимые переходы:
  - `new -> done` разрешён.
  - `paused -> done` разрешён.
- В `backend/internal/task/usecase_test.go`:
  - добавлен table-driven тест `TestUpdateTaskStatus_ToDone_FromAnyActiveStatus` для `new`, `in_progress`, `paused` -> `done`;
  - негативный тест переведён на реально невалидный кейс `new -> paused`.
- В `backend/internal/task/handler_test.go` добавлен `TestHandler_UpdateTaskStatus_ToDone_Success` (HTTP 204 для `PATCH .../status` с `status=done`).

**Tests / Verification**
- `cd backend && go test ./...`:
  - PASS (`ok improve-platform/internal/task 0.873s`; общий прогон `./...` успешен).
- Manual API smoke:
  - NOT RUN (рекомендуется проверить `PATCH /api/v1/tasks/{taskID}/status` для `new/in_progress/paused -> done` в интеграционном окружении).

**Impact**
- Изменение локализовано в модуле `task`:
  - доменная валидация переходов статуса;
  - unit/handler тесты.
- Формат API, коды ошибок вне контекста перехода в `done`, схема БД и миграции не изменялись.
- Обратная совместимость: да, за исключением намеренного расширения бизнес-правила переходов в `done`.

**Risks & Rollback**
- Риск: низкий; изменена только матрица переходов и тестовое покрытие в `task` модуле.
- Rollback:
  - вернуть прежние значения `validTransitions` в `backend/internal/task/usecase.go`;
  - удалить/скорректировать добавленные тесты для `-> done` из `new` и `paused`.
