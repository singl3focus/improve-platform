## TSK-0015 — Нормализовать ошибки невалидных UUID в API path-параметрах
- Status: Finished
- Type: Technical
- Approval: Approved
- Created: 2026-03-15
- Updated: 2026-03-15

### Context
- Ряд backend endpoint-ов принимает UUID в path-параметрах (`topicID`, `taskID`, `materialID`).
- При передаче невалидного идентификатора ошибка уходит в общий fallback и может возвращаться как `500 internal_error`.
- Это ухудшает UX и диагностику: клиентская ошибка должна возвращаться как `400 Bad Request` с явным кодом валидации.

### Acceptance Criteria
- [x] Для endpoint-ов с UUID path-параметрами добавлена валидация формата UUID до вызова usecase/repo.
- [x] Для невалидного UUID API возвращает `400 Bad Request` и код ошибки валидации (`validation_error`), без `internal_error`.
- [x] Поведение для валидных UUID (успехи и доменные ошибки `not_found/conflict`) не изменено.
- [x] Добавлены/обновлены handler-тесты, покрывающие кейсы невалидных UUID.
- [x] `go test ./...` в `backend/` проходит успешно.

### Constraints
- Не менять схему БД и миграции.
- Не менять бизнес-правила доменов roadmap/task/material.
- Изменения должны быть локализованы в transport/handler-слое и тестах.

### Change Summary
**Problem**
- Path UUID-параметры (`topicID`, `taskID`, `materialID` и связанные идентификаторы в roadmap) не валидировались в handler до вызова `svc/repo`.
- При невалидном UUID ошибка могла уходить в общий fallback и превращаться в `500 internal_error`.

**Root Cause Hypothesis**
- Отсутствовала ранняя проверка UUID в transport/handler-слое для path-параметров.
- Ошибки нижних слоёв по невалидному UUID не маппились в пользовательскую `validation_error`.

**Evidence**
- Валидация добавлена в handler-слой до вызова `svc`:
  - `backend/internal/task/handler.go:63`, `backend/internal/task/handler.go:216`
  - `backend/internal/material/handler.go:63`, `backend/internal/material/handler.go:164`
  - `backend/internal/roadmap/handler.go:136`, `backend/internal/roadmap/handler.go:395`
- Добавлены/обновлены handler-тесты с path params и кейсами невалидных UUID:
  - `backend/internal/task/handler_test.go:243`, `backend/internal/task/handler_test.go:270`
  - `backend/internal/material/handler_test.go:306`, `backend/internal/material/handler_test.go:333`
  - `backend/internal/roadmap/handler_test.go:267`, `backend/internal/roadmap/handler_test.go:294`, `backend/internal/roadmap/handler_test.go:321`
- Команда проверки:
  - `go test ./...` → PASS (включая `internal/roadmap`, `internal/task`, `internal/material`)

**Solution**
- Добавлена проверка формата UUID regex-паттерном в `roadmap/task/material` handlers.
- Для невалидных path UUID добавлен ранний ответ `400 Bad Request` с кодом `validation_error` и сообщением `<field> must be a valid UUID`.
- В существующих handler-тестах, где читаются path params через `chi.URLParam`, добавлен `chi.RouteContext` с корректными UUID.
- Добавлены новые тесты на невалидные UUID, проверяющие `400 validation_error` и отсутствие вызова service.

**Tests / Verification**
- Запущено: `go test ./...` в `backend/`.
- Результат: PASS.
- Ручной smoke API не запускался локально в рамках задачи.

**Impact**
- API: изменено только поведение ошибок для невалидных path UUID (ранний `400 validation_error` вместо потенциального fallback `500`).
- Данные/миграции: без изменений.
- Безопасность: без изменений.
- Обратная совместимость: да, для валидных UUID и доменных ошибок (`not_found/conflict`) поведение сохранено.

**Risks & Rollback**
- Риск: если внешний клиент передаёт не-UUID идентификаторы в path (вне контракта), теперь получит `400` раньше.
- Rollback plan: откатить изменения в `backend/internal/roadmap/handler.go`, `backend/internal/task/handler.go`, `backend/internal/material/handler.go` и соответствующие `handler_test.go`.

**Suggested Commit Message**
- `backend: return bad request for malformed uuid path parameters`
