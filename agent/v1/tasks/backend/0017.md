## TSK-0017 — Автосинхронизация статуса темы с чек-листом задач (`paused` = в работе)
- Status: Finished
- Type: Business
- Approval: Approved
- Created: 2026-03-16
- Updated: 2026-03-16

### Context
- Сейчас возможен конфликт: тема в `not_started`, но задачи ее чек-листа уже в `in_progress`.
- По требованию owner `paused` тоже считается “в работе” и не должен оставлять тему в `not_started`.
- Нужна серверная бизнес-логика, чтобы статус темы автоматически отражал факт реальной работы по чек-листу.

### Acceptance Criteria
- [ ] Если тема в `not_started` и хотя бы один checklist-task в `in_progress`, `paused` или `done`, тема автоматически переводится в `in_progress`.
- [ ] Существующий автопереход темы в `completed` при полном выполнении checklist сохраняется и покрыт тестами.
- [ ] Переход `not_started -> in_progress` из-за активности задач работает при обновлении статуса задач через API.
- [ ] Добавлены/обновлены unit-тесты модуля `task` и интеграционные проверки переходов статуса темы.
- [ ] `cd backend && go test ./...` проходит успешно.

### Constraints
- Не менять схему БД и миграции.
- Не менять публичный API-контракт статусов задач и тем.
- Изменения локализовать в бизнес-логике backend (`task`/`roadmap` взаимодействие и тесты).

### Change Summary

**Problem**
- Тема могла оставаться в `not_started`, когда её checklist-задачи уже были в `in_progress`, `paused` или `done`. Статус темы не отражал реальную активность по чек-листу.

**Root Cause Hypothesis**
- Автосинхронизация статуса темы работала только в одном направлении: `in_progress → completed` (при завершении всех задач). Обратный переход `not_started → in_progress` при начале работы над задачами отсутствовал.

**Evidence**
- `backend/internal/task/usecase.go`, функция `UpdateTaskStatus` (строки 176–180): `tryAutoCompleteTopic` вызывалась только при `req.Status == "done"`.
- `backend/internal/task/model.go`, интерфейс `TopicStatusUpdater`: содержал только `GetTopicStatus` и `SetTopicCompleted`, без метода для перевода в `in_progress`.

**Solution**
1. Добавлен метод `SetTopicInProgress(ctx, topicID, userID) error` в интерфейс `TopicStatusUpdater` (`task/model.go`).
2. Реализован `SetTopicInProgress` в адаптере `topicStatusAdapter` (`server/server.go`) — вызывает `repo.UpdateTopicStatus` со статусом `in_progress` и `startDate = today`.
3. Добавлена функция `tryAutoStartTopic` в `task/usecase.go`: проверяет текущий статус темы, и если он `not_started` — переводит в `in_progress` с записью history-события (`trigger: auto_start`).
4. В `UpdateTaskStatus` добавлен вызов `tryAutoStartTopic` при переходе checklist-задачи в `in_progress`, `paused` или `done`. Вызов стоит перед `tryAutoCompleteTopic`, чтобы корректно обработать цепочку `not_started → in_progress → completed` при единственной задаче, сразу ставшей `done`.
5. Добавлены 5 unit-тестов в `usecase_test.go`. Обновлён существующий тест `TestUpdateTaskStatus_ToDone_NotAllDone` (добавлен `getTopicStatusFn`).

**Tests / Verification**
- Запущено: `cd backend && go test ./... -v -count=1`
- Результат: все тесты прошли (exit code 0), включая:
  - `TestUpdateTaskStatus_ToInProgress_AutoStartTopic` — задача `new → in_progress`, тема `not_started` → тема `in_progress`
  - `TestUpdateTaskStatus_ToPaused_AutoStartTopic` — задача `in_progress → paused`, тема `not_started` → тема `in_progress`
  - `TestUpdateTaskStatus_ToDone_AutoStartAndCompleteTopic` — задача `new → done`, единственная задача, тема `not_started` → `in_progress` → `completed`
  - `TestUpdateTaskStatus_ToInProgress_TopicAlreadyInProgress` — тема уже `in_progress` → `SetTopicInProgress` НЕ вызывается
  - `TestUpdateTaskStatus_ToInProgress_StandaloneTask_NoAutoStart` — задача без `TopicID` → авто-старт не срабатывает
- Все существующие тесты (auto-complete, overdue, progress) продолжают проходить.

**Impact**
- Затронутые модули: `task` (бизнес-логика + тесты), `server` (адаптер).
- API-контракт не изменён (ни эндпоинты, ни формат запросов/ответов).
- Обратная совместимость: да (новая логика добавляет поведение, не меняет существующее).

**Risks & Rollback**
- Риск: минимальный. Логика идемпотентна — если тема уже не `not_started`, ничего не происходит.
- Откат: вернуть 4 файла к предыдущей версии (`task/model.go`, `task/usecase.go`, `task/usecase_test.go`, `server/server.go`).
