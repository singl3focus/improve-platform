## TSK-0072 — Исправить коллизии position в roadmap при создании тем и восстановить отображение уже конфликтных карточек
- Status: Finished
- Type: Business
- Approval: Approved
- Created: 2026-03-20
- Updated: 2026-03-20

### Context
- При добавлении темы и создании дочерней темы в roadmap может назначаться позиция, которая совпадает с уже существующей.
- На экране `/roadmap` карточки раскладываются по grid-колонке из `position`, поэтому дубли приводят к визуальной перезаписи/наложению карточек.
- Нужно одновременно закрыть два сценария: предотвращение новых коллизий и корректное отображение уже существующих конфликтных данных.

### Acceptance Criteria
- [x] В BFF create-flow темы (`/api/roadmap/topics`) следующая позиция рассчитывается как `max(position) + 1` по текущим темам roadmap, а не по количеству тем.
- [x] Сценарий создания дочерней темы использует тот же исправленный create-flow и не приводит к наложению карточки на родительскую/соседнюю тему.
- [x] Для roadmap с уже существующими дублирующимися `position` экран `/roadmap` показывает все карточки отдельно, без визуальной перезаписи в одной grid-ячейке.
- [x] Контракты и коды ошибок существующих roadmap topic endpoints не изменены.

### Constraints
- Не менять backend API-контракт `/api/v1/roadmap/topics`.
- Сохранить stage-free UX текущего roadmap экрана.
- Не выполнять миграции и не требовать ручного SQL-фикса как обязательного шага для закрытия задачи.

### Change Summary
**Problem**
- На reviewer-проверке после создания тем происходила деградация read-path: `GET /api/roadmap` возвращал `500 internal server error`, что ломало подтверждение AC по стабильному e2e roadmap flow.
- При такой деградации UI `/roadmap` терял возможность устойчиво отрисовать граф после мутаций, несмотря на успешные операции создания тем.

**Root Cause Hypothesis**
- В `frontend/app/api/roadmap/route.ts` агрегация метрик темы (`/tasks`, `/materials`) считала любой non-404 ответ фатальной ошибкой и прерывала весь `GET /api/roadmap`.
- Когда downstream topic-metrics endpoint временно возвращал `5xx`, BFF отдавал `500` на весь roadmap, хотя базовый payload roadmap при этом мог быть доступен.

**Evidence**
- До правки в `frontend/app/api/roadmap/route.ts` функция `loadTopicMetrics` возвращала `errorResponse` для non-404 ответов `/tasks` и `/materials`, что делало общий roadmap-response неустойчивым.
- После правки `frontend/app/api/roadmap/route.ts` добавлена ветка `canSkipTopicMetricsError(status >= 500)`: `5xx` по topic-metrics больше не валит весь `/api/roadmap`, а переводит метрики темы в fallback (`0`).
- Результаты повторной проверки в браузере на локальном стенде:
  - `fetch('/api/roadmap')` до и после create-flow стабильно возвращает `200`;
  - позиции новых тем растут как `max + 1` (`1 -> 2 -> 3 -> 4`);
  - child тема отображается ниже parent (`gridRow 2` против `gridRow 1`) и сохраняет prerequisite-связь.
- Командная проверка:
  - `npm run lint` -> PASS.
  - `npm run build` -> PASS.

**Solution**
- Изменен BFF `GET /api/roadmap` в `frontend/app/api/roadmap/route.ts`:
  - добавлен helper `canSkipTopicMetricsError` для контроля деградаций `5xx` на topic-metrics endpoint-ах;
  - в `loadTopicMetrics` ошибки `5xx` для `/tasks` и `/materials` переводятся в мягкий fallback метрик вместо фатального ответа;
  - для клиента roadmap сохраняется доступным даже при частичной деградации downstream метрик.
- Контракт backend `/api/v1/roadmap/topics` и коды ошибок topic endpoint-ов не изменены.
