## TSK-0079 — Ручная смена статуса темы в модалке редактирования roadmap
- Status: Finished
- Type: Business
- Approval: Approved
- Created: 2026-03-20
- Updated: 2026-03-21 (Reviewer)

### Context
- Сейчас статус темы визуализируется в карточке, но не управляется напрямую из модалки редактирования темы.
- Требуется добавить ручное изменение статуса в edit-flow roadmap с соблюдением бизнес-ограничений по допустимым переходам.

### Acceptance Criteria
- [x] В модалке редактирования темы добавлено поле статуса (из допустимых значений домена).
- [x] Сохранение модалки отправляет изменение статуса через корректный BFF/API flow для roadmap topic status.
- [x] UI не позволяет обходить доменные правила: недопустимые переходы показывают ошибку и не применяются.
- [x] Для заблокированной темы запрещено вручную устанавливать `in_progress` и `completed`; пользователь получает понятное сообщение.
- [x] Добавлены/обновлены frontend-тесты для успешной и неуспешной смены статуса из edit modal.

### Constraints
- Поддерживать только допустимые переходы статусов (без произвольных override).
- Не менять логику задач/чеклистов, которая уже может влиять на автопереходы темы.
- Не использовать git-команды.

### Change Summary
**Problem**
- В roadmap edit modal изначально нельзя было вручную менять статус темы: UI показывал статус только в карточке, а submit-flow отправлял только `title` и `description`.
- После первого фикса reviewer подтвердил поведение в браузере, но отклонил задачу по AC5: тесты покрывали только pure transition helper и не доказывали путь от поля `Статус` в edit modal до submit/BFF status-flow.

**Root Cause Hypothesis**
- В `frontend/src/components/roadmap-view.tsx` draft и submit edit modal не содержали поля `status`.
- В `frontend/app/api/roadmap/topics/[topicId]/route.ts` BFF не принимал `status` и не вызывал backend endpoint `/api/v1/roadmap/topics/{topicId}/status`.
- Во frontend не было общего helper-а, который зеркалит доменные правила переходов и blocked-guard из backend roadmap usecase.
- После первого фикса submit/status-логика оставалась внутри component/route, а тестовый контур `frontend/tsconfig.test.json` собирает только `src/lib`; из-за этого modal-flow и BFF status-flow не были изолированы в тестируемые unit-level точки.

**Evidence**
- Кодовая проверка: `frontend/src/components/roadmap-view.tsx` до изменения обновлял тему через `updateRoadmapTopic()` только с `title`/`description`; edit modal не рендерил control статуса.
- Кодовая проверка: `frontend/app/api/roadmap/topics/[topicId]/route.ts` до изменения проксировал только `PUT /api/v1/roadmap/topics/{topicId}` и не отправлял status-flow.
- Кодовая проверка: `backend/internal/roadmap/usecase.go` содержит доменные переходы `not_started -> in_progress`, `in_progress -> paused/completed`, `paused -> in_progress/not_started`, `completed -> in_progress` и блокировку ручного перехода blocked-темы в `in_progress`/`completed`.
- Reviewer note в этой задаче зафиксировал gap: отсутствовали тесты на `frontend/src/components/roadmap-view.tsx` submit-flow и на BFF status-flow в `frontend/app/api/roadmap/topics/[topicId]/route.ts`.
- Кодовая проверка после `ReMake`: `frontend/src/components/roadmap-view.tsx` теперь использует `prepareRoadmapTopicEditSubmission()` из `frontend/src/lib/roadmap-topic-edit.ts`, а BFF route использует `buildRoadmapTopicUpdatePlan()` из `frontend/src/lib/roadmap-topic-update-flow.ts`.
- `cd frontend && npm run test` → PASS (`49`/`49`).
- Ручной browser smoke на review-стенде повторно подтвердил success-path и оба error-path из edit modal.

**Solution**
- Добавлен общий helper `frontend/src/lib/roadmap-topic-status.ts` с проверкой доменных переходов и blocked-guard; добавлены unit-тесты в `frontend/src/lib/roadmap-topic-status.test.ts`.
- Для `ReMake` добавлен helper `frontend/src/lib/roadmap-topic-edit.ts`; `handleTopicUpdate` в `frontend/src/components/roadmap-view.tsx` теперь берёт из него готовый modal submit result (`payload`, `title_required`, `invalid_transition`, `blocked`), а не держит submit-ветвление только внутри компонента.
- Для `ReMake` добавлен helper `frontend/src/lib/roadmap-topic-update-flow.ts`; `frontend/app/api/roadmap/topics/[topicId]/route.ts` теперь строит через него единый BFF plan для `PATCH /status` + `PUT /topic` и для error-response веток.
- Добавлены тесты `frontend/src/lib/roadmap-topic-edit.test.ts` и `frontend/src/lib/roadmap-topic-update-flow.test.ts`, которые закрывают reviewer-gap по `success`, `invalid transition` и `blocked transition`.
- Для ручной локальной проверки обновлён `frontend/scripts/roadmap-tasks-ui-review-backend.mjs`: добавлены `GET/PUT/PATCH` topic routes и динамический пересчёт blocked-state по зависимостям.

**Tests / Verification**

- `cd frontend && npm run build` — PASS (вне sandbox из-за `spawn EPERM` в sandbox).

**Impact**
- Изменение затрагивает roadmap topic edit-flow в frontend/BFF, два новых testable helper-а в `src/lib` и локальный review-backend script для ручной верификации.
- Логика задач, checklist и автопереходов темы не менялась.
- Обратная совместимость: да, существующий edit-flow для `title`/`description` сохранён.

**Risks & Rollback**
- Риск: modal/BFF helper-ы могут разойтись с доменными правилами backend в будущем; снижен тем, что оба helper-а продолжают опираться на единый `roadmap-topic-status` transition-map, а BFF всё равно вызывает backend `/status` endpoint.
- Rollback: вернуть изменения в `frontend/src/components/roadmap-view.tsx`, `frontend/app/api/roadmap/topics/[topicId]/route.ts`, удалить `frontend/src/lib/roadmap-topic-edit.ts`, `frontend/src/lib/roadmap-topic-edit.test.ts`, `frontend/src/lib/roadmap-topic-update-flow.ts`, `frontend/src/lib/roadmap-topic-update-flow.test.ts`, откатить изменения в `frontend/tsconfig.test.json`, убрать topic routes из `frontend/scripts/roadmap-tasks-ui-review-backend.mjs`.
