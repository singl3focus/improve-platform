- Owner указал, что stage-концепт должен быть убран из frontend flow roadmap.

### Change Summary
**Problem**
- В roadmap UI оставались stage-остатки: stage-tag на карточках и stage-тексты в create-flow.
- В topic create/edit формах UI требовал ручной ввод `position`, а BFF topic routes зависели от `position` в клиентском payload.

**Root Cause Hypothesis**
- Логика create/edit тем и BFF маршруты были реализованы с историческим stage/position-centric подходом и не были полностью упрощены после перехода к stage-free пользовательскому сценарию.

**Evidence**
- `frontend/src/components/roadmap-view.tsx`: присутствовали `roadmap-topic-stage-tag`, `topicFieldPosition`, `topicCreateDraft.position`, `topicEditDraft.position`, `parsePositiveInteger(...)` в create/update flow.
- `frontend/app/api/roadmap/topics/route.ts`: был парсинг и валидация `payload.position`.
- `frontend/app/api/roadmap/topics/[topicId]/route.ts`: update требовал `payload.position` и возвращал 422 при его отсутствии.
- `frontend/scripts/mvp-verification-smoke.mjs`: stage-free topic create/update/dependency сценарии отправляли `position` в payload.

**Solution**
- В UI roadmap:
  - удален stage-tag из карточек тем;
  - удалены поля `position` из модалок create/edit темы;
  - удалены state/handlers/валидации, связанные с ручным `position` в create/edit.
- В BFF roadmap topics:
  - create route больше не читает `position` из клиентского payload, position вычисляется server-side fallback логикой;
  - update route больше не требует `position` от клиента, использует текущую позицию темы из backend GET как fallback.
- В smoke-проверке roadmap:
  - topic create/update/dependency сценарии обновлены на stage-free payload без `position` от клиента.

**Impact**
- Topic create/edit UX в `/roadmap` упрощен: без ручного `position`.
- Карточки roadmap визуально очищены от stage-лейблов при сохранении статусов/прогресса/зависимостей.
- BFF topic routes стали stage-free по входному контракту клиента (`position` не обязателен).
- Обратная совместимость: да (backend по-прежнему получает `position` из BFF fallback логики).

**Risks & Rollback**
- Риск: если backend ожидает специфическое клиентское управление позицией, порядок новых тем будет полностью определяться fallback логикой BFF.
- Rollback: вернуть чтение/валидацию `position` в topic BFF routes и соответствующие поля в roadmap create/edit формах.
