## TSK-0017 — Декомпозировать крупные frontend view-компоненты на hooks и UI-блоки
- Status: Finished
- Type: Technical
- Approval: Approved
- Created: 2026-03-15
- Updated: 2026-03-15

### Context
- Ключевые экраны содержат крупные монолитные компоненты со смешением fetch/mutation/state/render логики: `materials-library-view.tsx` (722), `dashboard-view.tsx` (500), `tasks-kanban-view.tsx` (421), `topic-workspace-view.tsx` (423).
- Такая структура замедляет поддержку, усложняет локальные изменения и повышает риск регрессий при доработках.
- Требуется структурный рефакторинг: выделить hooks и презентационные блоки, сохранив текущее поведение.

### Acceptance Criteria
- [x] Для `materials`, `dashboard`, `tasks`, `topics` экранов логика работы с данными вынесена из page-level view в отдельные hooks/сервисные модули.
- [x] UI-компоненты разделены на более мелкие блоки с понятной ответственностью (без изменения пользовательских сценариев).
- [x] Поведение экранов и контракты вызовов `/api/*` не изменились.
- [x] `npm run lint`, `npm run build`, `npm run verify:mvp` проходят успешно.

### Constraints
- Не менять backend API.
- Не делать редизайн и не менять UX-потоки в рамках задачи.
- Избегать массового переименования файлов без необходимости.

### Change Summary
**Problem**
- После декомпозиции view-компонентов не хватало проверяемых доказательств по `roadmap/topics`: в smoke-проверке не было явных success/error/fallback сценариев для этих экранов, поэтому AC по неизменности поведения не подтверждался.

**Root Cause Hypothesis**
- `frontend/scripts/mvp-verification-smoke.mjs` покрывал auth/tasks/materials/dashboard, но не покрывал `roadmap` и `topics` маршрутами, которые использует UI.

**Evidence**
- В `Reviewer Notes` зафиксировано, что `verify:mvp` не доказывал поведение `topics/roadmap`.
- После доработки `npm run verify:mvp` показывает PASS по новым сценариям:
  - `Roadmap GET route keeps response shape`
  - `Roadmap GET route keeps empty fallback when backend returns roadmap_not_found`
  - `Topics GET route keeps success and fallback payload shapes`
  - `Topics checklist PATCH route keeps success and key error statuses`

**Solution**
- Расширен `frontend/scripts/mvp-verification-smoke.mjs`:
  - добавлен mock-control endpoint `POST /__mock/scenario` в in-memory backend скрипта;
  - добавлен режим `roadmapMode` для эмуляции backend ответа `404 roadmap_not_found`;
  - добавлены проверки `roadmap` success/fallback;
  - добавлены проверки `topics` success/fallback и `topics checklist PATCH` (успех + ключевые ошибки `422/400/404`).
- Существующие проверки auth/tasks/materials/dashboard сохранены без изменения контрактов.

**Tests / Verification**
- `npm run lint` — PASS.
- `npm run build` — PASS (после повторного запуска вне sandbox из-за `spawn EPERM` в sandbox).
- `npm run verify:mvp` — PASS, включая новые `roadmap/topics` сценарии.

**Impact**
- Усилено автоматизированное доказательство неизменности поведения для `dashboard`, `roadmap`, `topics`, `tasks`, `materials`.
- Контракты `/api/*` и backend API не изменены.
- Обратная совместимость: да (изменен только verification smoke-скрипт).

**Risks & Rollback**
- Риск: низкий; изменение локализовано в smoke-скрипте проверки.
- Rollback plan:
  - вернуть `frontend/scripts/mvp-verification-smoke.mjs` к предыдущей версии,
  - повторно запустить `npm run verify:mvp`.
