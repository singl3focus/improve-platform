## TSK-0018 — Устранить `internal server error` при открытии `/topics` без `topicId`
- Status: Finished
- Type: Business
- Approval: Approved
- Created: 2026-03-15
- Updated: 2026-03-15

### Context
- При открытии `/topics` без query-параметра `topicId` frontend использует fallback demo-id `topic-html-css`.
- Backend ожидает UUID-идентификаторы тем, поэтому запрос с demo-id приводит к серверной ошибке и пользователь видит `internal server error`.
- Нужно убрать невалидный fallback и сделать предсказуемое поведение экрана при отсутствии выбранной темы.

### Acceptance Criteria
- [x] Переход на `/topics` без `topicId` не приводит к `500/internal server error`.
- [x] Экран `/topics` без `topicId` не отправляет запрос с demo-id; вместо этого показывает корректное пустое состояние с явной подсказкой выбрать тему в roadmap.
- [x] Поведение `/topics?topicId=<валидный-id>` не регрессирует: рабочее пространство темы загружается как раньше.
- [x] Ошибки загрузки темы остаются пользовательскими и диагностируемыми (без падения страницы).
- [x] `npm run lint`, `npm run build`, `npm run verify:mvp` проходят успешно после изменений.

### Constraints
- Не менять backend API в рамках этой задачи.
- Не возвращать fallback на hardcoded demo-id.
- Не менять бизнес-логику чеклиста/материалов темы; только сценарий входа в экран без выбранной темы.

### Change Summary
**Problem**
- При открытии `/topics` без `topicId` компонент отправлял запрос с fallback `topic-html-css`, что приводило к backend-ошибке и `internal server error` на экране.

**Root Cause Hypothesis**
- В `TopicWorkspaceView` использовался hardcoded fallback demo-id вместо явного сценария "тема не выбрана".
- `useTopicWorkspaceViewModel` всегда запускал загрузку темы, потому что принимал только `string` и не поддерживал idle-состояние.

**Evidence**
- `frontend/src/components/topic-workspace-view.tsx:341` — `topicId` теперь нормализуется как `searchParams.get("topicId")?.trim() || null` (без fallback demo-id).
- `frontend/src/components/hooks/use-topic-workspace-view-model.ts:69` — хук принимает `topicId: string | null`.
- `frontend/src/components/hooks/use-topic-workspace-view-model.ts:78` — при отсутствии `topicId` выставляется `idle` и загрузка не запускается.
- `frontend/src/components/hooks/use-topic-workspace-view-model.ts:91` — fetch выполняется только с валидным `currentTopicId`.
- Проверки:
  - `npm run lint` → PASS.
  - `npm run build` → PASS.
  - `npm run verify:mvp` → PASS.

**Solution**
- Убран fallback demo-id в `TopicWorkspaceView`; отсутствие `topicId` теперь трактуется как отдельный сценарий.
- Добавлено пустое состояние на `/topics` без `topicId` с явной подсказкой и кнопкой перехода в `/roadmap`.
- Добавлены RU/EN тексты для пустого состояния.
- Обновлён `useTopicWorkspaceViewModel`: добавлено состояние `idle`, отключён fetch при `topicId = null`.
- Добавлены стили для блока пустого состояния в `frontend/app/globals.css`.

**Tests / Verification**
- Запущено `npm run lint` — PASS.
- Запущено `npm run build` — PASS.
- Запущено `npm run verify:mvp` — PASS.
- Ручной smoke `/topics` без query-параметров не запускался в браузере в рамках текущей сессии.

**Impact**
- Затронут только frontend-сценарий входа на `/topics` без выбранной темы.
- Backend API и контракты не изменялись.
- Обратная совместимость: да (`/topics?topicId=<valid-id>` продолжает работать через прежний flow загрузки).

**Risks & Rollback**
- Риск: при ошибочном формировании ссылки без `topicId` пользователь увидит пустое состояние вместо попытки загрузки темы.
- Rollback: вернуть изменения в `frontend/src/components/topic-workspace-view.tsx`, `frontend/src/components/hooks/use-topic-workspace-view-model.ts`, `frontend/app/globals.css` к предыдущей версии.
