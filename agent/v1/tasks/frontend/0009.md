## TSK-0009 — Рефакторинг mock API-роутов: единая обработка режимов ответа
- Status: Finished
- Type: Technical
- Approval: Approved
- Created: 2026-03-14
- Updated: 2026-03-14

### Context
- В mock API-роутах frontend повторяется одинаковая логика обработки `mode` (`default|empty|error|forbidden|conflict`) и генерации `NextResponse.json`.
- Дублирование присутствует в `roadmap`, `topics`, `tasks`, `materials` route handlers и повышает риск расхождения контрактов/сообщений ошибок при изменениях.
- Нужен точечный рефактор без изменения бизнес-поведения: вынести общее поведение в единый helper и подключить его в релевантных endpoint.

### Acceptance Criteria
- [x] Для mock API роутов `roadmap/topics/tasks/materials` (включая мутационные роуты) используется единый helper для чтения `mode` и формирования ответов `403/409/500/empty`.
- [x] Текущие HTTP-коды и payload сообщений для поддерживаемых режимов остаются обратно-совместимыми с текущим поведением (`verify:mvp` без регрессий по edge-cases).
- [x] Удалено дублирование локальных веток `if (mode === "...")` в целевых роутах, кроме endpoint-specific валидации.
- [x] Проект проходит `npm run lint`, `npm run build`, `npm run verify:mvp`.

### Constraints
- Не менять публичный API-контракт mock-эндпоинтов и структуру успешных payload.
- Не смешивать с интеграцией на live backend `/api/v1/*` в рамках этой задачи.
- Изменения ограничить frontend API-слоем и утилитами, которые напрямую поддерживают эти роуты.
- Избегать косметического рефакторинга вне зоны дублирования mode-handling.

### Change Summary
**Problem**
- В mock API-роутах (`roadmap/topics/tasks/materials`) повторялась одинаковая логика `mode` и однотипные `NextResponse.json` для `error/empty/forbidden/conflict`.

**Root Cause Hypothesis**
- В каждом endpoint mode-handling был реализован локально через ручные `if (mode === "...")` без общего helper, поэтому дублирование разрасталось на GET/PATCH/POST/DELETE роуты.

**Evidence**
- Дублирование mode-веток было в целевых роутах: `frontend/app/api/roadmap/route.ts`, `frontend/app/api/topics/[topicId]/route.ts`, `frontend/app/api/tasks/route.ts`, `frontend/app/api/materials/route.ts` и мутационных роутах `topics/.../checklist`, `tasks/[taskId]/status`, `materials/[materialId]`.
- После рефакторинга проверка `rg -n "if \\(mode === \\\"(error|empty|forbidden|conflict)\\\"\\)" frontend/app/api` не нашла совпадений.
- Контрактная проверка `npm run verify:mvp` прошла: все smoke-сценарии PASS, включая пути `403/409` и edge-cases мутаций.

**Solution**
- Добавлен единый helper `frontend/src/lib/mock-api-mode.ts`:
  - `getMockApiMode(request)` для чтения и нормализации `mode`.
  - `getMockModeResponse(request, options)` для стандартных ответов `500/200(empty)/403/409` с endpoint-specific сообщениями.
- Целевые роуты переведены на helper без изменения успешных payload и текстов сообщений:
  - `frontend/app/api/roadmap/route.ts`
  - `frontend/app/api/topics/[topicId]/route.ts`
  - `frontend/app/api/topics/[topicId]/checklist/[itemId]/route.ts`
  - `frontend/app/api/tasks/route.ts`
  - `frontend/app/api/tasks/[taskId]/status/route.ts`
  - `frontend/app/api/materials/route.ts`
  - `frontend/app/api/materials/[materialId]/route.ts`

**Tests / Verification**
- `npm run lint` — PASS.
- `npm run build` — PASS.
- `npm run verify:mvp` — PASS (сервер поднят на `http://127.0.0.1:3025`).
- `rg`-проверка на mode-ветки — PASS (совпадений нет).

**Impact**
- API: публичный контракт mock-эндпоинтов сохранён (HTTP-коды и payload-сообщения для поддерживаемых mode unchanged).
- Данные: изменений форматов/хранилища нет.
- Безопасность: изменений auth/secrets/ACL нет.
- Обратная совместимость: да.

**Risks & Rollback**
- Риск: при дальнейшем изменении helper возможно затронуть сразу несколько mock-роутов.
- Rollback: удалить использование `getMockModeResponse` из затронутых роутов и вернуть локальные mode-ветки; при необходимости удалить `frontend/src/lib/mock-api-mode.ts`.
