## TSK-0008 — Проверка корректности frontend-реализации относительно backend
- Status: Finished
- Type: Technical
- Approval: Approved
- Created: 2026-03-14
- Updated: 2026-03-14

### Context
- Владелец запросил отдельную задачу проверки корректности frontend со своей стороны.
- Нужна воспроизводимая проверка реальных MVP-сценариев, а не ручные предположения.
- Проверка должна учитывать появившиеся визуальные референсы в `design/`, не подменяя ими функциональные требования.

### Acceptance Criteria
- [x] Зафиксирован список критичных пользовательских MVP-сценариев frontend (auth, roadmap, topic, tasks, materials, dashboard).
- [x] Для сценариев реализованы e2e/smoke-проверки или эквивалентный автоматизированный набор.
- [x] Проверяется корректная обработка API-ошибок и edge-cases (403/404/validation/conflict) в UI.
- [x] Сформирован отчет по несовпадениям frontend ожиданий и backend контракта.
- [x] В отчет включена отдельная секция: где есть визуальные отклонения от `design/_1.._5/code.html` и почему они допустимы в рамках функциональных требований задач.

### Constraints
- Проверять только реально реализованные MVP-сценарии.
- Нельзя утверждать успех без запуска проверок и фиксированного результата.
- Проверки должны быть воспроизводимы локально.
- `design/*` использовать только как reference-only ориентир для визуальной проверки; функциональная корректность определяется задачами и backend-контрактом.

### Change Summary
**Problem**
- Не было единого воспроизводимого набора проверок критичных MVP-сценариев frontend.
- Не было автоматизированного отчета о рассинхронах между frontend ожиданиями и backend контрактом.
- Проверка edge-cases `403/404/validation/conflict` и допустимых визуальных отклонений от `design/_1.._5` не была формализована в одной процедуре.

**Root Cause Hypothesis**
- Проверки выполнялись точечно и вручную, без единой команды для прогона.
- Контрактная сверка frontend-vs-backend не была автоматизирована.
- В проекте не было отдельного скрипта, который одновременно проверяет MVP-потоки и фиксирует gaps.

**Evidence**
- Добавлен сценарный smoke-скрипт: `frontend/scripts/mvp-verification-smoke.mjs`.
- Добавлена команда запуска: `frontend/package.json` → `verify:mvp`.
- Добавлены явные `403/409` ветки в frontend API-слое:
  - `frontend/app/api/roadmap/route.ts`;
  - `frontend/app/api/topics/[topicId]/route.ts`;
  - `frontend/app/api/topics/[topicId]/checklist/[itemId]/route.ts`;
  - `frontend/app/api/tasks/route.ts`;
  - `frontend/app/api/tasks/[taskId]/status/route.ts`;
  - `frontend/app/api/materials/route.ts`;
  - `frontend/app/api/materials/[materialId]/route.ts`;
  - `frontend/app/api/dashboard/_shared.ts`.
- Фактический прогон `npm run verify:mvp` показал:
  - PASS по сценариям `auth guard`, `roadmap`, `topic workspace`, `tasks board`, `materials CRUD`, `dashboard widgets`, `explicit 403/409 paths`.
  - Выявленные contract gaps:
  - roadmap/topic/tasks/materials/dashboard во frontend реализованы как mock/in-memory API и не подключены к backend `/api/v1/*`;
  - vocabulary mismatch: frontend task status `completed`, backend task status `done`.

**Solution**
- Реализован автоматизированный smoke-набор `verify:mvp` с проверкой критичных пользовательских потоков:
  - `auth`: redirect guard + validation error.
  - `roadmap`: default/empty/error.
  - `topic`: load + checklist mutation + `404/422`.
  - `tasks`: list/filter + status mutation + rollback + `404/422`.
  - `materials`: list/create/update/delete + `404/422`.
  - `dashboard`: все 6 виджет-эндпоинтов + empty/error режимы.
- Добавлена реальная проверка edge-cases `403/409` в smoke:
  - `roadmap`, `topics`, `topics/checklist`, `tasks`, `tasks/status`, `materials`, `materials/[id]`, `dashboard/progress`.
- В API-роутах MVP добавлены явные ветки `mode=forbidden` (`403`) и `mode=conflict` (`409`) для воспроизводимой проверки.
- В тот же скрипт добавлен статический contract-diff frontend-vs-backend с автоматическим выводом несовпадений.
- Зафиксирована отдельная секция визуальных отклонений (см. ниже) и обоснование, почему это допустимо в рамках scope задач.

**Tests / Verification**
- Запущено:
  - `npm run lint` — PASS.
  - `npm run build` — PASS (sandbox: `spawn EPERM`, вне sandbox: PASS).
  - `npm run verify:mvp` при поднятом `next start --hostname 127.0.0.1 --port 3025` — PASS (8/8 сценариев).
- Не запускалось локально:
  - Runtime smoke на live backend с БД и реальными auth/session токенами.

**Visual Deviations (design/_1.._5)**
- `design/_1` (dashboard): в текущем UI нет декоративных маркетинговых блоков (премиум-card, цитата, медиа-аватары), сохранен функциональный MVP-фокус на сводке/прогрессе/задачах/блокерах/материалах.
- `design/_2` (roadmap): отсутствуют демо-элементы `share/filter/search` в header; сохранена ключевая функциональная часть (этапы, DAG-зависимости, blocked reasons, legend, переходы в тему).
- `design/_3` (topic): не реализованы community/footer/promo-блоки; сохранены рабочие блоки темы для MVP (hero, dependencies, checklist mutation, materials order).
- `design/_4` (tasks): отсутствуют декоративные quick actions и расширенный top nav; функциональный kanban с 4 статусами, overdue-индикатором и сменой статуса через API сохранен.
- `design/_5` (materials): нет медиа-thumbnail карточек и upgrade CTA; сохранены требуемые сценарии библиотеки (поиск/фильтры, CRUD, progress, позиционирование, переход к теме).
- Обоснование: `design/*` использован как visual reference-only; функциональные требования задач и backend-контракт сохранены приоритетом.

**Impact**
- Добавлен только tooling-слой проверки (`verify:mvp`), без изменения продуктового runtime-поведения экранов.
- Обратная совместимость: да.

**Risks & Rollback**
- Риск: smoke-команда зависит от доступности локального frontend сервера на ожидаемом порту.
- Риск: static contract-diff не заменяет end-to-end прогон с live backend и БД.
- Rollback:
  - удалить `frontend/scripts/mvp-verification-smoke.mjs`;
  - удалить `verify:mvp` из `frontend/package.json`.
