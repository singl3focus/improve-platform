## TSK-0039 — Рефакторинг roadmap frontend-модуля без изменения поведения
- Status: Finished
- Type: Technical
- Approval: Approved
- Created: 2026-03-16
- Updated: 2026-03-16

### Context
- Компонент roadmap-view содержит смешанную ответственность: загрузка данных, CRUD-мутаторы, dependency-операции, рендер графа и inline/edit UX в одном файле.
- Такая структура затрудняет сопровождение, повышает риск регрессий и усложняет локальное тестирование/ревью.
- Нужен ограниченный рефакторинг frontend-кода с сохранением текущей функциональности.

### Acceptance Criteria
- [x] Логика roadmap декомпозирована на подкомпоненты/хуки с разделением ответственности (graph/stage/topic/dependency/mutation state).
- [x] Публичное поведение roadmap не меняется: существующие сценарии загрузки, ошибок и CRUD остаются рабочими.
- [x] Удалены явные дублирования кода, упрощены обработчики состояний и мутаций.
- [x] `npm run lint` и `npm run build` в `frontend/` проходят успешно.

### Constraints
- Без изменения бизнес-правил roadmap и API-контрактов.
- Не вводить тяжелые внешние UI-библиотеки для рефакторинга.
- Любые визуальные изменения должны быть минимальными и только как следствие структурного рефакторинга.

### Change Summary
**Problem**
- `src/components/roadmap-view.tsx` содержал одновременно graph layout-логику, состояние draft-форм и крупные inline JSX-блоки для stage/topic/dependency/quick-create, что затрудняло сопровождение.

**Root Cause Hypothesis**
- Основная причина сложности: отсутствие явной декомпозиции по зонам ответственности, из-за чего обработчики и формы были тесно связаны внутри одного компонента.

**Evidence**
- До изменений ключевая логика графа и все mutation-формы были инлайн в `frontend/src/components/roadmap-view.tsx` (один компонент `RoadmapView`).
- После изменений в `frontend/src/components/roadmap-view.tsx`:
  - выделен hook `useRoadmapGraphLayout` для расчета связей/размера графа;
  - выделены подкомпоненты `StageMutationPanel`, `TopicMutationPanel`, `DependencyMutationPanel`, `QuickCreatePanel`.
- Для закрытия ReMake добавлен воспроизводимый артефакт функциональной проверки: `frontend/scripts/mvp-verification-smoke.mjs` расширен roadmap-сценариями (loading/error + stage/topic/dependency CRUD), включая поддержку соответствующих mock backend endpoint-ов.
- Результат smoke-запуска подтверждён локально: все новые roadmap-сценарии завершились PASS.
- Проверка командами:
  - `node scripts/mvp-verification-smoke.mjs` -> PASS;
  - `npm run lint` -> PASS;
  - `npm run build` -> PASS.

**Solution**
- Добавлен hook `useRoadmapGraphLayout`, в который перенесен расчет SVG-связей и размеров графа.
- Добавлены подкомпоненты форм для stage/topic/dependency/quick-create, чтобы отделить рендер форм от контейнерной orchestration-логики.
- В `RoadmapView` добавлены локальные draft-update handlers (`handleStageDraftTitleChange`, `handleTopicCreateStageChange`, и др.), что устранило повторяющиеся inline `setState`-блоки.
- Сохранены текущие API-вызовы, валидации, сообщения об ошибках и последовательность `roadmap.reload()`.
- В рамках ReMake расширен `frontend/scripts/mvp-verification-smoke.mjs`:
  - добавлены roadmap smoke-сценарии для `GET /api/roadmap` (успех + API error), `POST /api/roadmap/quick-create`, stage CRUD, topic CRUD и dependency add/remove;
  - добавлена поддержка соответствующих mock backend маршрутов (`/api/v1/roadmap/stages/:id`, `/api/v1/roadmap/topics/:id`, `/api/v1/roadmap/topics/:id/dependencies`, `/api/v1/roadmap/topics/:id/dependencies/:dependencyId`) для воспроизводимой проверки без изменения production API-контрактов.

**Tests / Verification**
- `cd frontend && node scripts/mvp-verification-smoke.mjs` -> PASS (все roadmap сценарии из замечания reviewer: loading/error/CRUD).
- `cd frontend && npm run lint` -> PASS.
- `cd frontend && npm run build` -> PASS.

**Impact**
- API: production API без изменений; расширен только test/mock контур в smoke-скрипте.
- Данные: без изменений.
- UI-поведение CRUD/ошибок/загрузки: без функциональных изменений.
- Обратная совместимость: да.

**Risks & Rollback**
- Риск: регрессия в связке пропсов подкомпонентов форм.
- Риск: изменения в smoke-моке могут устаревать при эволюции backend контракта.
- Rollback: вернуть `frontend/scripts/mvp-verification-smoke.mjs` к предыдущей версии (убрать добавленные roadmap CRUD smoke-сценарии и mock handlers).
