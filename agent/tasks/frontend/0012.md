## TSK-0012 — Добавить обязательное поле `full_name` в форму регистрации и frontend auth-контракты
- Status: Finished
- Type: Business
- Approval: Approved
- Created: 2026-03-15
- Updated: 2026-03-15

### Context
- Owner подтвердил, что `full_name` обязателен на регистрации.
- Сейчас frontend регистрация отправляет только `email/password`; поле имени отсутствует в форме, типах и BFF парсинге.
- Без этой доработки backend изменение по `full_name` не сможет быть использовано из UI.

### Acceptance Criteria
- [x] В UI регистрации (`/register`) добавлено поле `full_name` и оно обязательно для отправки формы.
- [x] `frontend/src/lib/auth/contracts.ts` и клиентские типы/запросы регистрации расширены полем `full_name`.
- [x] `frontend/app/api/auth/_shared.ts` (`parseAuthCredentials`) валидирует и передает `full_name` в backend auth register.
- [x] Ошибки валидации `full_name` корректно отображаются в существующем error-flow формы.
- [x] После успешной регистрации пользовательская сессия сохраняется как и раньше (без регрессий login/register flow).

### Constraints
- Не менять URL-контракты auth-роутов (`/api/auth/*` и backend `/api/v1/auth/*`).
- Не ослаблять текущую валидацию email/password.
- Поле `full_name` в frontend payload должно совпадать с backend контрактом (snake_case).
- Не смешивать с задачей персонализации dashboard.

### Change Summary
**Problem**
- Регистрация на frontend отправляла только `email/password`; поле `full_name` отсутствовало в UI, контрактах и BFF-валидации.
- Из-за этого frontend не мог передать обязательное поле `full_name` в backend register-контракт.

**Root Cause Hypothesis**
- В `AuthForm` отсутствовал input `full_name` для режима `register`.
- Тип регистрации использовал общий `AuthCredentials` без `full_name`.
- `parseAuthCredentials` в BFF валидировал только `email/password` одинаково для login/register.

**Evidence**
- `frontend/src/components/auth/auth-form.tsx`: добавлено обязательное поле `name="full_name"` и отправка `full_name` в `register(...)`.
- `frontend/src/lib/auth/contracts.ts`: добавлен `AuthRegisterCredentials` с обязательным `full_name`.
- `frontend/src/lib/auth/client.ts`: `register(credentials: AuthRegisterCredentials)`.
- `frontend/app/api/auth/_shared.ts`: `parseAuthCredentials(..., { requireFullName: true })`-совместимая валидация и передача `full_name`.
- `frontend/app/api/auth/register/route.ts`: register-роут вызывает `parseAuthCredentials(request, { requireFullName: true })`.
- Команды проверки:
  - `npm run lint` -> PASS.
  - `npm run build` -> PASS (после повторного запуска вне sandbox).

**Solution**
- Добавлено обязательное поле `full_name` в форму регистрации (`/register`) с отправкой в snake_case payload.
- Расширены frontend auth-контракты отдельным типом регистрации с обязательным `full_name`.
- Обновлен auth-клиент: register-запрос теперь типизирован `AuthRegisterCredentials`.
- Расширен BFF-парсер auth payload:
  - добавлена опция `requireFullName`;
  - для register-ветки `full_name` обязателен и передается в backend `/api/v1/auth/register`;
  - для login-ветки поведение сохранено (`email/password`).
- Текст валидационной ошибки для register-ветки: `Full name, email and password are required.`; ошибка возвращается через существующий auth error-flow.

**Tests / Verification**
- Запущено:
  - `npm run lint` — PASS.
  - `npm run build` — PASS (`Compiled successfully`, `Generating static pages (26/26)`).
- Техническая оговорка:
  - Первый `npm run build` в sandbox завершился `spawn EPERM`; повторный запуск вне sandbox прошёл успешно.
- Не запускалось:
  - Ручной e2e регистрации в браузере (`/register` без/с `full_name`) не выполнялся локально.

**Impact**
- API-контракты: frontend register payload расширен обязательным `full_name` (snake_case), URL auth-роутов не менялись.
- UI: на `/register` появился обязательный input `full_name`.
- Auth flow: login-поток не изменен; сохранение сессии после успешной register/login-ответа осталось прежним.
- Обратная совместимость: частичная — backend register теперь ожидается совместимым с `full_name` (для нового контракта это целевое поведение).

**Risks & Rollback**
- Риск: если backend register-контракт еще не принимает `full_name`, регистрация будет возвращать ошибку backend.
- Риск: ручной e2e без живого backend пока не подтвержден.
- Rollback plan:
  - убрать `full_name` из `AuthForm` и register payload;
  - вернуть прежнюю сигнатуру register-клиента и `parseAuthCredentials` без `requireFullName`;
  - вернуть вызов register-route к общей валидации без `full_name`.
