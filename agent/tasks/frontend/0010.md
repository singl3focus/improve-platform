## TSK-0010 — Исправить Not Found при регистрации через строгий auth-префикс `/api/v1`
- Status: Finished
- Type: Business
- Approval: Approved
- Created: 2026-03-15
- Updated: 2026-03-15

### Context
- Пользовательский сценарий регистрации падает с `Not Found`.
- Расследование показало расхождение контрактов: frontend BFF отправляет auth-запросы на `/auth/*`, а backend обслуживает только `/api/v1/auth/*`.
- Для устранения дефекта нужно выровнять frontend-контракт под фактический backend API без добавления legacy-совместимости.

### Acceptance Criteria
- [x] В frontend auth BFF все auth-пути используют строгий префикс `/api/v1/auth/*` для `register/login/refresh/logout`.
- [x] Server-side проверка сессии во frontend использует `/api/v1/auth/refresh` (без legacy-путей `/auth/*`).
- [x] Сценарий регистрации через UI (`/register`) больше не возвращает `Not Found` при доступном backend и валидном payload.
- [x] Поведение login/refresh/logout после правки не регрессирует относительно текущего контракта cookie-сессии.

### Constraints
- Следовать только строгой стратегии API: использовать только `/api/v1/*`.
- Не добавлять fallback/alias на `/auth/*` во frontend.
- Не менять бизнес-логику backend и формат токенов/куки в рамках этой задачи.

### Change Summary
**Problem**
- Регистрация через frontend BFF могла получать `404 Not Found`, потому что backend auth-эндпоинты доступны только под `/api/v1/auth/*`, а frontend запрашивал `/auth/*`.

**Root Cause Hypothesis**
- Несогласованный контракт путей между frontend и backend: в auth BFF и server-side проверке сессии использовались legacy-пути `/auth/*`.

**Evidence**
- В `frontend/app/api/auth/_shared.ts` были заданы пути `/auth/login`, `/auth/register`, `/auth/refresh`, `/auth/logout`.
- В `frontend/src/lib/auth/server-session.ts` `hasValidServerSession` вызывал `/auth/refresh`.
- После правки в обоих файлах используется строгий префикс `/api/v1/auth/*`.

**Solution**
- Обновлён `BACKEND_AUTH_PATHS` в `frontend/app/api/auth/_shared.ts`:
  - `login` -> `/api/v1/auth/login`
  - `register` -> `/api/v1/auth/register`
  - `refresh` -> `/api/v1/auth/refresh`
  - `logout` -> `/api/v1/auth/logout`
- Обновлён server-side refresh-check в `frontend/src/lib/auth/server-session.ts`:
  - `/auth/refresh` -> `/api/v1/auth/refresh`

**Tests / Verification**
- `npm run lint` — PASS.
- `npm run build` — PASS (после перезапуска вне sandbox).
- Ручной `/register` e2e-сценарий — PASS (`GET /register -> 200`, `POST /api/auth/register -> 200`, `404` не воспроизводится).
- Проверка login/refresh/logout + cookie-сессии — PASS (`login -> 200`, `session -> 200`, `refresh -> 200`, `logout -> 200` + `Set-Cookie Max-Age=0`, `session after logout -> 401`).

**Impact**
- API: frontend auth BFF теперь соответствует backend-контракту `/api/v1/auth/*`.
- Данные: без изменений в форматах токенов/куки.
- Безопасность: изменений auth-модели, ACL и secret-management нет.
- Обратная совместимость: с backend, который обслуживает только `/api/v1/auth/*`, совместимость улучшена; legacy `/auth/*` намеренно не поддерживается.

**Risks & Rollback**
- Риск: при backend без `/api/v1/auth/*` auth-сценарии не будут работать (ожидаемое поведение по текущему контракту).
- Rollback: вернуть значения `BACKEND_AUTH_PATHS` и путь в `hasValidServerSession` к прежним `/auth/*`.

**Suggested Commit Message**
- `frontend: switch auth bff routes to /api/v1 and fix register not-found`
