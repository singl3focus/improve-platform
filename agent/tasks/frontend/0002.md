### Context
- Для MVP нужны пользовательские сессии и изоляция доступа к личным данным.

- [x] Реализованы экраны регистрации и логина по email/password.

- `design/*` не является источником функциональных требований auth-flow; ориентироваться на acceptance criteria задачи и backend-контракт.

### Change Summary
**Problem**
- Во frontend отсутствовал реализованный auth-flow: не было экранов логина/регистрации, единого API-клиента для auth-контрактов, хранения сессии и route guards для приватных разделов.
- После первого ReMake оставался дефект: при невалидных cookie на `/login` middleware без проверки валидности сессии редиректил на `/dashboard`, что приводило к циклу `/login -> /dashboard -> /login`.
- После следующего ReMake оставался дефект локального MVP-режима: auth cookies выставлялись с `Secure` на HTTP origin, из-за чего после успешной регистрации `GET /api/auth/session` в той же сессии возвращал `401`.

**Root Cause Hypothesis**
- Frontend-каркас для MVP auth ещё не был реализован.
- Не было BFF-слоя `/api/auth/*` для безопасной работы с токенами через HttpOnly cookies.
- Не было общего формата обработки и отображения auth-ошибок.
- В `frontend/middleware.ts` ветка `isAuthRoute(pathname) && hasSession` использовала только факт наличия cookie, без валидации через `/api/auth/session`.
- Cookie policy в `frontend/app/api/auth/_shared.ts` определяла `secure` только через `NODE_ENV === "production"`, не учитывая реальный протокол запроса (`http`/`https`).

**Evidence**
- В `frontend/` до изменений были только `package.json` и `next-env.d.ts`, без `app/` и auth-модулей.
- Централизованный auth API-клиент добавлен в `frontend/src/lib/auth/client.ts`.
- Route guards и приватные роуты добавлены в `frontend/middleware.ts` и `frontend/app/(private)/layout.tsx`.
- UI auth-ошибок унифицирован через `frontend/src/components/auth/api-error-alert.tsx` и используется в `frontend/src/components/auth/auth-form.tsx`.
- После ReMake:

  - `/api/auth/session` в `frontend/app/api/auth/session/route.ts` больше не считает access-cookie валидной сессией без проверки refresh.
  - защита от open redirect вынесена в `frontend/src/lib/auth/next-path.ts` и используется в `AuthForm`.
- После текущего ReMake:

  - при невалидных cookie сценарий `login?next=/dashboard` завершился `HTTP 200` без redirect loop.
- После текущего ReMake #3:

  - auth route handlers (`login/register/refresh/session/logout`) передают `request` в cookie helper-функции;
  - на локальном HTTP (`127.0.0.1:3012`) регистрация сразу поднимает рабочую сессию (`/api/auth/session` -> `200`).

**Solution**
- Добавлен минимальный App Router каркас frontend с публичными и приватными маршрутами (`frontend/app/*`).
- Реализованы страницы логина и регистрации на email/password (`frontend/app/login/page.tsx`, `frontend/app/register/page.tsx`) с общей формой `AuthForm`.
- Добавлен централизованный auth API-клиент (`frontend/src/lib/auth/client.ts`) и единый парсинг API-ошибок (`frontend/src/lib/auth/api-error.ts`).
- Добавлен BFF-слой для auth-запросов (`frontend/app/api/auth/*`) с проксированием на backend и хранением access/refresh токенов в HttpOnly cookies.
- Реализован refresh-механизм сессии (`/api/auth/refresh`, `/api/auth/session`, `fetchWithSessionRefresh`).
- Добавлены route guards:
  - edge middleware redirect для неавторизованных пользователей из приватных роутов;
  - server-side guard в приватном layout.
- Для устранения цикла редиректов обновлена edge-логика auth-страниц:
  - редирект `/login|/register -> /dashboard` выполняется только при подтвержденной валидной сессии;
  - при невалидной сессии cookie очищаются, и страница логина/регистрации остается доступной.
- Для устранения дефекта local HTTP сессии обновлена cookie policy в auth BFF:
  - `Secure` выставляется только для HTTPS-запросов (включая reverse-proxy сценарий через `x-forwarded-proto`);
  - в local/dev HTTP режиме cookies остаются HttpOnly/SameSite=Lax, но без `Secure`, чтобы браузер/клиент мог применить их на `http://127.0.0.1:*`.

**Impact**
- API: добавлен frontend BFF-контур `/api/auth/*` для auth-контрактов.
- Данные: токены сессии хранятся в HttpOnly cookies на стороне frontend-сервера; чувствительные данные не выводятся в UI/логи.
- Безопасность: приватные разделы защищены middleware и server-side guard; устранен redirect-loop при невалидной сессии на auth-страницах.
- Безопасность: в production HTTPS policy не ослаблена (`Secure` сохраняется), локальный HTTP MVP-режим перестал терять сессию после регистрации.
- Обратная совместимость: да (изменения изолированы в frontend, backend-контракт не меняется).

**Risks & Rollback**
- Риск: backend может иметь другой контракт auth-эндпоинтов/полей токенов.
- Риск: server-side guard сейчас основан на проверке `/auth/refresh`; при недоступном backend приватные роуты всегда редиректят на `/login`.
- Риск: middleware теперь делает дополнительный запрос к `/api/auth/session` для auth-страниц при наличии cookie.
- Риск: если reverse proxy не передаёт корректный `x-forwarded-proto`, определение HTTPS может опираться только на `request.nextUrl.protocol`.
- Rollback plan:
  - удалить добавленные frontend auth-модули и вернуть `agent/tasks/frontend/0002.md` к предыдущему состоянию задачи;
  - либо вернуть в `frontend/middleware.ts` прежнюю ветку редиректа auth-роутов без session-check;
  - либо откатить изменения сигнатур cookie helper-функций и вернуть статическое правило `secure: NODE_ENV === "production"` в `frontend/app/api/auth/_shared.ts`.
