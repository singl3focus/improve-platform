## TSK-0026 — Статус задач в теме только для просмотра (read-only)
- Status: Finished
- Type: Business
- Approval: Approved
- Created: 2026-03-16
- Updated: 2026-03-16

### Context
- В `Topic Workspace` статус checklist-задачи сейчас меняется через `select`.
- По требованию owner в просмотре темы статус не должен редактироваться, должен отображаться только текущий.
- Управление статусами задач должно оставаться в профильном разделе `/tasks`.

### Acceptance Criteria
- [x] На экране `/topics?topicId=...` статус каждого checklist-item отображается как read-only badge/text без `select`.
- [x] Из `Topic Workspace` больше не отправляются `PATCH`-запросы на изменение статуса checklist-item.
- [x] Текущий статус задач корректно подтягивается из backend и отображается без потери локализации.
- [x] Изменение статусов в `/tasks` продолжает отражаться в `Topic Workspace` после обновления данных.

### Constraints
- Не менять backend API и статусную модель задач.
- Не переносить логику редактирования статуса в другие экраны кроме `/tasks`.
- Изменения ограничить `Topic Workspace` и связанным view-model.

### Change Summary
**Problem**
- В `Topic Workspace` чеклист показывал `select` и позволял менять статус задачи прямо на `/topics`, что нарушает требование read-only для этого экрана.

**Root Cause Hypothesis**
- Компонент `TopicChecklistPanel` рендерил интерактивный `select`, а view-model `use-topic-workspace-view-model` содержал `PATCH`-мутацию статуса (`patchChecklistStatus` + `handleChecklistChange`).

**Evidence**
- `frontend/src/components/topic-workspace-view.tsx`: в `TopicChecklistPanel` использовался `select` с `onChecklistChange`.
- `frontend/src/components/hooks/use-topic-workspace-view-model.ts`: присутствовал `fetch(..., { method: "PATCH" })` на `/api/topics/{topicId}/checklist/{itemId}`.
- Проверка после изменений: `cd frontend && npm run lint` (PASS), `cd frontend && npx tsc --noEmit` (PASS).

**Solution**
- Удалена мутация статуса из `use-topic-workspace-view-model`: удалены `patchChecklistStatus`, `handleChecklistChange`, а также `updatingItemId`/`updateError`.
- В `TopicChecklistPanel` `select` заменен на read-only badge с локализованным текстом статуса.
- Добавлен маппинг класса бейджа для checklist-статусов (`todo`/`in_progress`/`done`) с существующими status-токенами UI.
- Обновлен subtitle чеклиста: статусы изменяются в `/tasks`, а не в `Topic Workspace`.

**Tests / Verification**
- `cd frontend && npm run lint` — PASS (`next lint`: no warnings/errors).
- `cd frontend && npx tsc --noEmit` — PASS (exit code 0).
- Manual — NOT RUN; рекомендован сценарий: изменить статус в `/tasks`, обновить `/topics?topicId=...`, убедиться, что в теме отображается актуальный статус.

**Impact**
- API: `Topic Workspace` больше не отправляет `PATCH` по checklist-status; используется только чтение актуального состояния.
- Данные: формат и модель статусов без изменений.
- Безопасность: изменений нет.
- Обратная совместимость: да.

**Risks & Rollback**
- Риск: если пользователи ожидали редактирование статуса из `/topics`, поведение изменилось на read-only по требованиям.
- Rollback: вернуть `select` и обработчик `onChecklistChange` в `topic-workspace-view`, а также `patchChecklistStatus`/`handleChecklistChange` в `use-topic-workspace-view-model`.
