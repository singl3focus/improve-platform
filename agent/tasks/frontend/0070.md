### Change Summary
**Problem**
- На карточке задачи в kanban отсутствовало действие редактирования, а также отображалась лишняя drag-подсказка, создающая визуальный шум.
- После первичной реализации оставались блокеры ReMake: сценарий `topic -> no topic` в edit-модалке не применялся, а create-flow `/tasks` на стенде ревью периодически завершался `500 internal server error`.
- В текущей итерации ReMake сохранялся риск нестабильности create-flow: подготовительный шаг вычисления `position` через `GET /api/v1/tasks` мог блокировать создание задачи целиком при временной ошибке backend.

**Root Cause Hypothesis**
- В UI `tasks-kanban-view.tsx` карточка имела только кнопку удаления и не имела модального сценария редактирования.
- В BFF-роуте `frontend/app/api/tasks/[taskId]/route.ts` был реализован только `DELETE`, без `PUT` для обновления задачи.
- В ремейке дополнительно выявлено:
  - backend task update (`backend/internal/task/model.go`, `backend/internal/task/repo.go`, `backend/internal/task/usecase.go`) фактически не применял `topic_id` при `PUT /api/v1/tasks/{taskID}`;
  - BFF create-flow отправлял создание без вычисленной позиции, из-за чего на окружениях со строгим учётом `position` был риск `500`.
- В текущей итерации дополнительно: подготовительный `GET /api/v1/tasks` в BFF create-flow имел fail-fast поведение (любой не-OK ответ прерывал `POST /api/v1/tasks`), что делало создание хрупким при кратковременных ошибках на шаге расчёта позиции.

**Evidence**
- До изменения карточка рендерила `<p className="tasks-card-drag-hint">...</p>` и не содержала edit-кнопки.
- До изменения `frontend/app/api/tasks/[taskId]/route.ts` содержал только обработчик `DELETE`.
- До ремейка в update-потоке использовались условные конструкции `...(topicId ? { topicId } : {})` и `...(topicId ? { topic_id: topicId } : {})`, что отбрасывало `no topic`.
- До текущей доработки backend `UpdateRequest` не содержал `topic_id`, а SQL `UPDATE tasks` не обновлял поле `topic_id`.
- До текущей доработки create BFF (`frontend/app/api/tasks/route.ts`) возвращал ошибку `Failed to prepare task creation.` и не выполнял `POST /api/v1/tasks`, если предварительный `GET /api/v1/tasks` возвращал не-OK.
- После изменений:
  - `Set-Location frontend; npm run lint` → PASS.
  - `npm run build` (из `frontend`) → PASS.
  - `Set-Location backend; go test ./internal/task/...` → PASS.

**Solution**
- В `frontend/src/components/tasks-kanban-view.tsx`:
  - добавлена отдельная кнопка редактирования в каждой карточке задачи;
  - добавлена модалка редактирования с полями `title`, `description`, `topic`, `deadline`;
  - реализован поток закрытия модалки только после успешного сохранения;
  - при ошибке API модалка остается открытой и показывает ошибку;
  - удален drag-hint текст из карточки.
- В `frontend/src/components/hooks/use-tasks-board-view-model.ts`:
  - добавлен `handleUpdate` и клиентский вызов `PUT /api/tasks/{taskId}`;
  - обновление конкретной карточки в state после успешного ответа.
- В ремейке: `topicId` в update-запросе отправляется явно как `topicId || null`, чтобы поддержать снятие привязки темы.
- В `frontend/app/api/tasks/[taskId]/route.ts`:
  - добавлен `PUT` handler, проксирующий обновление через backend-контракт `PUT /api/v1/tasks/{taskID}`;
  - после PUT выполняется повторный `GET` задачи и формируется ответ в формате kanban-карточки.
- В ремейке: в backend body поле `topic_id` передается всегда (включая `null`), чтобы работали оба направления: `no topic -> topic` и `topic -> no topic`.
- В `frontend/app/globals.css`:
  - добавлены стили для группы действий карточки и кнопки edit (включая dark theme).
- В `backend/internal/task/model.go` добавлена поддержка `topic_id` в `UpdateRequest` с tri-state поведением (`не передан` / `строка` / `null`) через custom `UnmarshalJSON`.
- В `backend/internal/task/usecase.go` и `backend/internal/task/repo.go` добавлено условное обновление `topic_id` в `PUT /api/v1/tasks/{taskID}`.
- В `frontend/app/api/tasks/route.ts` create-flow усилен:
  - перед созданием вычисляется `position` (`max + 1`) по текущему списку задач;
  - в backend payload `topic_id` передается явно (включая `null`), вместе с рассчитанной `position`.
- В текущей итерации `frontend/app/api/tasks/route.ts` дополнительно усилен fallback create-flow:
  - вычисление `position` вынесено в `resolveTaskCreatePosition`;
  - при неуспешном ответе предварительного `GET /api/v1/tasks` используется безопасный fallback `position = 1`, и создание задачи продолжается через `POST /api/v1/tasks`.
