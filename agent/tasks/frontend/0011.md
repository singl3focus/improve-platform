## TSK-0011 — Полноценный аккаунт: убрать mock-seed и подключить приватные экраны к backend данным
- Status: Finished
- Type: Business
- Approval: Approved
- Created: 2026-03-15
- Updated: 2026-03-15

### Context
- Сейчас приватные экраны показывают предзаполненные данные из frontend mock/in-memory источников, из-за чего новый аккаунт выглядит как уже заполненный.
- Это противоречит ожидаемому поведению: новый пользователь после регистрации должен видеть пустое состояние до создания своих сущностей.
- Нужна полноценная интеграция frontend BFF-роутов с backend `/api/v1/*` на основных пользовательских экранах без hardcoded demo payload.

### Acceptance Criteria
- [x] `frontend/app/api/roadmap/route.ts` получает данные из backend `/api/v1/roadmap` (через авторизованный BFF flow), а не из `roadmapPayload`.
- [x] `frontend/app/api/topics/[topicId]/route.ts` и `frontend/app/api/topics/[topicId]/checklist/[itemId]/route.ts` работают через backend (`/api/v1/roadmap/topics/{topicID}` и `.../tasks` / статусные мутации), без `topic-workspace-store`.
- [x] `frontend/app/api/tasks/route.ts` и `frontend/app/api/tasks/[taskId]/status/route.ts` работают через backend `/api/v1/tasks*`, без `tasks-board-store`.
- [x] `frontend/app/api/materials/route.ts` и `frontend/app/api/materials/[materialId]/route.ts` работают через backend `/api/v1/materials*`, без `materials-library-store`.
- [x] Dashboard API-роуты (`frontend/app/api/dashboard/*`) используют реальные backend-данные (агрегация в BFF допустима), а не константные `*Payload`.
- [x] Для нового пользователя без созданных roadmap/topics/tasks/materials UI всех приватных экранов показывает пустые состояния (не demo-данные).

### Constraints
- Не ломать текущий auth cookie-контракт frontend (`/api/auth/*` + `improve_access_token`/`improve_refresh_token`).
- Не использовать hardcoded бизнес-данные в API-роутах приватной зоны.
- Сохранить совместимость с backend статусами задач (`done`) и не оставлять рассинхрон `completed` vs `done`.
- Не смешивать с доработкой `full_name`, локализацией и темизацией в этой задаче.

### Change Summary
**Problem**
- Приватные BFF-роуты frontend возвращали demo/in-memory данные, поэтому новый аккаунт видел заполненные сущности вместо пустых состояний.

**Root Cause Hypothesis**
- Источником данных для приватной зоны были mock payload/store (`roadmapPayload`, `topic-workspace-store`, `tasks-board-store`, `materials-library-store`, dashboard `*Payload`) вместо backend `/api/v1/*`.

**Evidence**
- До изменений роуты импортировали mock-store/payload:
  - `frontend/app/api/roadmap/route.ts`
  - `frontend/app/api/topics/[topicId]/route.ts`
  - `frontend/app/api/tasks/route.ts`
  - `frontend/app/api/materials/route.ts`
  - `frontend/app/api/dashboard/*`
- После изменений все перечисленные роуты ходят в backend `/api/v1/*` через авторизованный BFF helper `frontend/src/lib/backend-api.ts` с cookie-based refresh flow.
- Команды проверки:
  - `npm run lint` -> PASS
  - `npm run build` -> PASS
  - `docker compose -f build/docker-compose.yml -p improve-platform-remake up -d --build` -> PASS
  - `docker compose -f build/docker-compose.yml -p improve-platform-remake ps` -> PASS (backend/postgres healthy)
  - e2e `register -> login -> private pages + private API` -> PASS, ответы приватных BFF для нового аккаунта содержат только пустые состояния (без demo-данных)

**Solution**
- Добавлен общий backend-клиент для BFF:
  - `frontend/src/lib/backend-api.ts` (authorized requests, refresh токена, проксирование ошибок, обновление/очистка auth cookies).
  - `frontend/src/lib/backend-learning-contracts.ts` и `frontend/src/lib/backend-learning-mappers.ts` (контракты/mapping статусов `done` <-> `completed`, `todo/in_progress/done`).
- Полностью заменены mock-источники на backend в роутерах:
  - `frontend/app/api/roadmap/route.ts`
  - `frontend/app/api/topics/[topicId]/route.ts`
  - `frontend/app/api/topics/[topicId]/checklist/[itemId]/route.ts`
  - `frontend/app/api/tasks/route.ts`
  - `frontend/app/api/tasks/[taskId]/status/route.ts`
  - `frontend/app/api/materials/route.ts`
  - `frontend/app/api/materials/[materialId]/route.ts`
  - `frontend/app/api/dashboard/_shared.ts`
  - `frontend/app/api/dashboard/daily-summary/route.ts`
  - `frontend/app/api/dashboard/progress/route.ts`
  - `frontend/app/api/dashboard/in-progress-topics/route.ts`
  - `frontend/app/api/dashboard/blocked-topics/route.ts`
  - `frontend/app/api/dashboard/upcoming-tasks/route.ts`
  - `frontend/app/api/dashboard/recent-materials/route.ts`
- Для новых аккаунтов при отсутствии roadmap возвращаются пустые payload (`stages: []`, пустые списки dashboard/materials/tasks, fallback workspace для topic route), без demo-данных.

**Tests / Verification**
- `npm run lint` -> PASS.
- `npm run build` -> PASS.
- `docker compose -f build/docker-compose.yml -p improve-platform-remake up -d --build` -> PASS.
- `docker compose -f build/docker-compose.yml -p improve-platform-remake ps` -> PASS (`backend`/`postgres` healthy).
- Ручной e2e `register -> login -> /dashboard,/roadmap,/topics,/tasks,/materials` -> PASS; приватные BFF endpoint'ы возвращают пустые payload для нового пользователя без demo-данных.

**Impact**
- API/BFF: приватные frontend-роуты теперь backend-driven, совместимы с cookie auth-контрактом (`improve_access_token`/`improve_refresh_token`).
- Статусы задач: сохранена совместимость backend `done` через mapping в UI `completed`.
- Обратная совместимость: да, для UI-контрактов текущих страниц (`/dashboard`, `/roadmap`, `/topics`, `/tasks`, `/materials`) сохранены ожидаемые формы ответов.

**Risks & Rollback**
- Риск: повышенная нагрузка на backend из-за агрегирующих вызовов (особенно dashboard/materials/roadmap).
- Риск: backend-ограничение `PUT /api/v1/materials/{id}` не поддерживает смену `topic_id`; BFF возвращает 422 при попытке смены темы.
- Rollback:
  - Вернуть предыдущие версии BFF-роутов в `frontend/app/api/{roadmap,topics,tasks,materials,dashboard}/**`.
  - Удалить новые helper-файлы `frontend/src/lib/backend-api.ts`, `frontend/src/lib/backend-learning-contracts.ts`, `frontend/src/lib/backend-learning-mappers.ts`.
