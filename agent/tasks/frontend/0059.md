## TSK-0059 — Растянуть controls-блок материалов на всю ширину с приоритетом поля поиска
- Status: Finished
- Type: Business
- Approval: Approved
- Created: 2026-03-17
- Updated: 2026-03-17

### Context
- Owner требует изменить верхний controls-блок на экране материалов:
  - блок должен занимать всю доступную ширину;
  - поле поиска должно занимать большую часть блока (около 60%).
- Текущая раскладка filters/controls не гарантирует такой приоритет поиска.

### Acceptance Criteria
- [x] Верхний controls-блок материалов занимает 100% ширины контейнера.
- [x] Поле поиска занимает около 60% ширины controls-блока на desktop.
- [x] Фильтр по теме и кнопка создания выровнены в оставшемся пространстве без наложений.
- [x] На mobile раскладка адаптивна и не ломает ввод/кнопки.
- [x] Визуально не ухудшены отступы/ритм относительно остального экрана материалов.

### Constraints
- Не менять бизнес-логику фильтрации и CRUD материалов.
- Изменения ограничить materials UI и связанными стилями.
- Не использовать git-команды.

### Change Summary
**Problem**
- Верхний controls-блок экрана материалов не фиксировал приоритет ширины поля поиска, из-за чего desktop-раскладка не соответствовала требованию owner.

**Root Cause Hypothesis**
- В controls использовался `flex` + обертка filters, где ширина search/select зависела от контента и `min-width`, а не от целевого распределения пространства.

**Evidence**
- `frontend/src/components/materials-library-view.tsx` — controls-разметка была перестроена: search/topic вынесены в отдельные grid-элементы одного уровня с кнопкой создания.
- `frontend/app/globals.css` — `materials-library-controls` переведен на `grid-template-columns: minmax(0, 6fr) minmax(220px, 3fr) auto`, что даёт приоритет search-полю около 60% на desktop.
- `frontend/app/globals.css` — в mobile media-query controls переключается в `grid-template-columns: 1fr` для адаптивного стека.
- Команды:
  - `Set-Location frontend; npm run lint` -> PASS.
  - `Set-Location frontend; npm run build` -> PASS.
- UI-проверка в браузере:
  - на `/materials` `Search/Topic/Create material` отображаются в controls-блоке без наложений;
  - `Create material` открывает модальное окно;
  - для текущего viewport controls работают в single-column responsive режиме.

**Solution**
- В `MaterialsLibraryHeader` удалена промежуточная обертка filters и добавлены отдельные модификаторы:
  - `materials-filter-item-search`,
  - `materials-filter-item-topic`,
  - `materials-create-trigger`.
- Для controls-блока материалов введен grid-layout на desktop:
  - колонка поиска (6fr),
  - колонка topic-фильтра (3fr),
  - колонка кнопки создания (`auto`).
- Для mobile сохранена адаптивность через переключение controls в одну колонку.
- Логика фильтрации и CRUD материалов не менялась.

**Tests / Verification**
- Запущено `Set-Location frontend; npm run lint` — PASS.
- Запущено `Set-Location frontend; npm run build` — PASS.
- Ручная UI-проверка в браузере запущена:
  - full-width controls + desktop grid rules — PASS;
  - alignment topic/create без наложений — PASS;
  - mobile responsive (single-column) + интеракции input/select/button — PASS.

**Impact**
- Изменения ограничены materials UI (header controls) и связанными стилями.
- API, данные и бизнес-логика материалов не изменялись.
- Обратная совместимость: да.

**Risks & Rollback**
- Риск: desktop-приоритет `~60%` зафиксирован через CSS grid rule (6fr/3fr/auto) и UI smoke; на очень узких контейнерах включается mobile media-query, что ожидаемо меняет раскладку в 1 колонку.
- Rollback: вернуть изменения в `frontend/src/components/materials-library-view.tsx` и `frontend/app/globals.css`.
