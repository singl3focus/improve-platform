## TSK-0006 — Экран задач в формате личного Kanban
- Status: Finished
- Type: Business
- Approval: Approved
- Created: 2026-03-14
- Updated: 2026-03-14

### Context
- Для MVP задачи должны быть доступны в отдельном kanban-представлении.
- Просрочка отображается визуально, но не как отдельный статус/колонка.
- Для визуального направления использовать `design/_4/code.html` как референс kanban-композиции.

### Acceptance Criteria
- [x] Реализованы колонки `новая`, `в работе`, `на паузе`, `выполнена`.
- [x] Карточка задачи показывает: название, связанную тему (если есть), дедлайн, индикатор просрочки, краткое описание.
- [x] Перемещение задачи между колонками обновляет статус через API.
- [x] Просроченность визуально видна без выделения отдельной колонки `просрочено`.
- [x] Визуальная структура доски (header, колонки, карточки, индикаторы статусов/дедлайнов) согласована с `design/_4/code.html` без обязательного пиксель-перфект совпадения.

### Constraints
- UI должен оставаться легким и личным, без командных/enterprise паттернов.
- Изменение статуса должно быть устойчивым к сетевым ошибкам (rollback UI состояния).
- Поддерживать фильтрацию по теме и дедлайну только в пределах MVP.
- `design/_4/code.html` трактуется только как визуальный пример; функциональные правила определяются этой задачей и `docs/bussines/v1/domain.md`.
- Не расширять функциональный scope из-за декоративных/демо-элементов, присутствующих в референсе.

### Change Summary
**Problem**
- Экран `/tasks` был заглушкой: отсутствовал kanban с колонками статусов, карточками задач, фильтрацией и API-обновлением статусов.

**Root Cause Hypothesis**
- Не существовало выделенного UI-компонента kanban-доски и API-контрактов для `list/update` задач.
- Не было server-state слоя задач для воспроизводимого обновления статусов.
- В глобальных стилях отсутствовали классы для layout header/filters/columns/cards под kanban-структуру.

**Evidence**
- До изменений `frontend/app/(private)/tasks/page.tsx` рендерил placeholder.
- Kanban UI реализован в `frontend/src/components/tasks-kanban-view.tsx`:
  - конфигурация 4 колонок (`COLUMN_CONFIG`),
  - overdue-логика (`isOverdue`),
  - обработчик перемещения между колонками через API (`handleStatusChange`) с optimistic update и rollback.
- Типы и server-state задач добавлены:
  - `frontend/src/lib/tasks-board-types.ts`,
  - `frontend/src/lib/tasks-board-store.ts`.
- API добавлен:
  - `GET /api/tasks` (`frontend/app/api/tasks/route.ts`),
  - `PATCH /api/tasks/[taskId]/status` (`frontend/app/api/tasks/[taskId]/status/route.ts`).
- Runtime smoke подтвердил изменение статуса задачи через API (`new -> paused`).

**Solution**
- Подключен новый экран задач:
  - `frontend/app/(private)/tasks/page.tsx` теперь рендерит `TasksKanbanView`.
- Реализован `TasksKanbanView`:
  - колонки `new/in_progress/paused/completed`,
  - карточка задачи с `title`, `description`, `topic` (если есть), `dueAt`, overdue-индикатором,
  - фильтры по теме и дедлайну (`all/overdue/week`),
  - изменение статуса карточки через `PATCH` API.
- Для устойчивости к сетевым ошибкам добавлен optimistic update + rollback:
  - статус карточки обновляется локально сразу,
  - при ошибке API UI откатывается к предыдущему снимку и показывает ошибку.
- Добавлен in-memory store задач с функциями list/update и фильтрацией.
- Добавлены стили kanban-доски в `frontend/app/globals.css` (header, filters, columns, cards, mobile layout).

**Tests / Verification**
- Запущено:
  - `npm run lint` — PASS.
  - `npm run build` — PASS (после повторного запуска вне sandbox из-за `spawn EPERM`).
  - Runtime smoke API list/update задач — PASS (`task-1002: new -> paused`).
  - Runtime smoke API error-режима — PASS (`/api/tasks?mode=error` -> `500`).
- Не запускалось локально:
  - Ручной browser smoke в `npm run dev` с интерактивной проверкой перетаскивания/перемещения карточек на всех колонках.

**Impact**
- API: добавлены новые frontend endpoints для задач:
  - `GET /api/tasks` c фильтрами `topicId` и `due`,
  - `PATCH /api/tasks/[taskId]/status`.
- UI: `/tasks` теперь полноценная personal-kanban доска с 4 статус-колонками.
- Поведение: обновление статуса задачи идет через API и устойчиво к ошибкам (rollback).
- Обратная совместимость: да, изменения изолированы в модуле задач frontend.

**Risks & Rollback**
- Риск: текущая реализация использует in-memory server-state; изменения статусов не переживают перезапуск процесса.
- Риск: реальный backend-контракт по задачам может отличаться от mock API (формат полей/фильтров).
- Rollback plan:
  - вернуть `frontend/app/(private)/tasks/page.tsx` к placeholder-состоянию;
  - удалить `frontend/src/components/tasks-kanban-view.tsx`,
    `frontend/src/lib/tasks-board-types.ts`,
    `frontend/src/lib/tasks-board-store.ts`,
    `frontend/app/api/tasks/route.ts`,
    `frontend/app/api/tasks/[taskId]/status/route.ts`;
  - убрать task-kanban стили из `frontend/app/globals.css`.
