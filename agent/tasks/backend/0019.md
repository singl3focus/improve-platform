## TSK-0019 — Добавить агрегированные метрики тем в GET /api/v1/roadmap
- Status: Finished
- Type: Technical
- Approval: Approved
- Created: 2026-03-16
- Updated: 2026-03-16

### Context
- Текущий BFF roadmap получает базовый граф, а затем догружает метрики каждой темы отдельными запросами (`tasks/materials`), что создает N+1 паттерн.
- Для масштабирования графа (больше тем/связей) нужно отдавать ключевые метрики тем сразу в payload `GET /api/v1/roadmap`.

### Acceptance Criteria
- [x] `GET /api/v1/roadmap` возвращает для каждой темы агрегированные поля: `tasks_count`, `materials_count`, `progress_percent`.
- [x] Агрегаты считаются в backend с учетом `user_id`-изоляции и не нарушают текущие DAG-инварианты.
- [x] Для пустых данных возвращаются корректные нулевые значения метрик (без ошибок сериализации).
- [x] Обновлены unit/handler/usecase тесты roadmap-модуля для нового response-контракта.
- [x] `cd backend; go test ./...` проходит успешно.

### Constraints
- Не менять миграции и схему БД в рамках этой задачи.
- Не менять бизнес-правила статусов тем и зависимостей.
- Изменение контракта должно быть backward-compatible: существующие поля roadmap остаются доступными.

### Change Summary
**Problem**
- `GET /api/v1/roadmap` возвращал структуру графа без агрегированных метрик темы, из-за чего BFF/клиенту приходилось догружать `tasks/materials` отдельными запросами (N+1).

**Root Cause Hypothesis**
- В roadmap-модуле отсутствовал слой агрегации метрик по темам: ни модель ответа, ни `usecase/repo` не собирали `tasks_count`, `materials_count`, `progress_percent`.

**Evidence**
- До изменения `TopicResponse` в `backend/internal/roadmap/model.go` не содержал полей `tasks_count`, `materials_count`, `progress_percent`.
- В `backend/internal/roadmap/usecase.go` метод `GetFullRoadmap` собирал только roadmap/stages/topics/dependencies и не выполнял запрос агрегатов.
- После изменения добавлен метод `GetTopicMetricsByUserID` в `backend/internal/roadmap/repo.go` с user-scoped SQL-агрегацией по темам.

**Solution**
- Расширен контракт ответа темы:
  - `backend/internal/roadmap/model.go`: в `TopicResponse` добавлены поля `tasks_count`, `materials_count`, `progress_percent`.
  - Добавлен доменный тип `TopicMetrics` и метод репозитория `GetTopicMetricsByUserID(ctx, userID)`.
- Реализован единый агрегирующий запрос в `backend/internal/roadmap/repo.go`:
  - выборка всех тем пользователя (`topics t WHERE t.user_id = $1`);
  - LEFT JOIN подзапросов по `tasks` и `materials` (только того же `user_id`);
  - `COALESCE(..., 0)` для нулевых значений;
  - `progress_percent` считается как `done/total*100` по checklist-задачам темы.
- Интеграция в сборку roadmap:
  - `backend/internal/roadmap/usecase.go`: `GetFullRoadmap` запрашивает metrics map и передает в `assembleRoadmap`;
  - `buildTopicResponse` получает метрики и заполняет новые поля;
  - если метрик для темы нет, возвращаются нули.
- Обновлены тесты roadmap:
  - `backend/internal/roadmap/usecase_test.go`: добавлена поддержка нового метода repo-мока; расширен `TestGetFullRoadmap_WithBlockageInfo` проверкой метрик; добавлен `TestGetFullRoadmap_MetricsZeroValuesWhenMissing`.
  - `backend/internal/roadmap/handler_test.go`: `TestHandler_GetRoadmap_Success` проверяет сериализацию новых полей.

**Impact**
- API: `GET /api/v1/roadmap` теперь возвращает дополнительные поля для каждой темы:
  - `tasks_count`
  - `materials_count`
  - `progress_percent`
- Обратная совместимость: да, существующие поля ответа не удалялись и не переименовывались.
- Бизнес-правила DAG/статусов не изменены.

**Risks & Rollback**
- Риск: дополнительная агрегация в `GetFullRoadmap` добавляет SQL-нагрузку при больших roadmap (компенсируется устранением N+1 на стороне BFF).
- Риск: `progress_percent` использует целочисленное деление, что может отличаться от ожидаемого округления на клиенте.
- Rollback: откатить изменения в `backend/internal/roadmap/model.go`, `backend/internal/roadmap/repo.go`, `backend/internal/roadmap/usecase.go` и соответствующих тестах.
