## TSK-0011 — Общий docker-compose для локального запуска MVP
- Status: Finished
- Type: Technical
- Approval: Approved
- Created: 2026-03-14
- Updated: 2026-03-14 (Reviewer)

### Context
- Для MVP нужен единый воспроизводимый запуск окружения без ручной сборки сервисов.
- Владелец явно запросил задачу на общий `docker-compose` в `build/`.

### Acceptance Criteria
- [ ] В `build/docker-compose.yml` описан запуск `frontend`, `backend`, `postgres` и фонового процесса уведомлений (бот/воркер).
- [ ] Запуск `docker compose -f build/docker-compose.yml up -d --build` поднимает контейнеры, а `backend` проходит health-check.
- [ ] Добавлен шаблон env (`.env.example` или эквивалент) для локального старта без секретов в коде.

### Constraints
- Self-hosted only, без cloud-only зависимостей.
- Конфигурация только через environment variables.
- Не использовать `git`-операции.

### Verification
- Command/Test:
  - `docker compose -f build/docker-compose.yml config`
  - Expected: валидная конфигурация без ошибок.
  - Result: PASS (выполнено с JWT_SECRET; config выводит services: backend, frontend, postgres без ошибок).
  - `docker compose -f build/docker-compose.yml up -d --build`
  - Expected: сервисы запущены, `backend` и `postgres` в состоянии healthy/running.
  - Result: NOT RUN (локально не запускалось; рекомендовано: скопировать build/.env.example в .env в корне репо, задать JWT_SECRET, выполнить команду; для работы backend при первом запуске применить миграции к БД отдельно).

### Change Summary
**Problem**
- Нет единого способа поднять MVP-окружение (frontend, backend, postgres) одной командой; ручная сборка и настройка.

**Root Cause Hypothesis**
- Отсутствовали общий docker-compose и шаблон env для стека.

**Evidence**
- Запрос в задаче: «общий docker-compose в build/», «шаблон env для локального старта».
- Файлы: `backend/internal/config/config.go` (env-переменные), `backend/cmd/server/main.go` (scheduler в процессе backend), `frontend/package.json` (Next.js), health: `/readyz` в `backend/internal/server/handler/health.go`.

**Solution**
- Добавлен `build/docker-compose.yml`: сервисы `postgres` (PostgreSQL 16-alpine, healthcheck), `backend` (сборка из backend.Dockerfile, healthcheck по /readyz, env из переменных окружения), `frontend` (сборка из frontend.Dockerfile, BACKEND_API_URL=http://backend:8080). Фоновый процесс уведомлений (Telegram) реализован в том же процессе backend при заданных TELEGRAM_BOT_TOKEN/TELEGRAM_CHAT_ID.
- Добавлены `build/backend.Dockerfile` (multi-stage: Go 1.25-alpine → alpine + wget для healthcheck) и `build/frontend.Dockerfile` (multi-stage: Node 20, next build + npm run start).
- Добавлен `build/.env.example`: все переменные для локального старта (POSTGRES_*, JWT_SECRET, порты, опционально Telegram); без секретов в коде.

**ReMake (по замечаниям ревьюера)**
- **Problem**: Сборка frontend падала на `build/frontend.Dockerfile:18`: `COPY --from=builder /app/public ./public` — каталог `/app/public` отсутствует в проекте (в frontend нет папки `public`).
- **Solution**: В builder-стадии `build/frontend.Dockerfile` добавлено `RUN mkdir -p public` после `COPY . .`, чтобы каталог существовал перед `npm run build` и run-стадия могла скопировать `public` без ошибки.
- **AC#1 (воркер уведомлений)**: Принятый контракт — фоновый процесс уведомлений (бот/воркер) запускается в том же процессе backend; отдельного сервиса в docker-compose нет. Доказательство: при заданных TELEGRAM_BOT_TOKEN/TELEGRAM_CHAT_ID backend при старте запускает scheduler (см. `backend/cmd/server/main.go`); проверка — логи backend при старте или успешный health-check после `docker compose up`.

**Tests / Verification**
- Выполнено: `docker compose -f build/docker-compose.yml config` (с JWT_SECRET) — конфигурация валидна (PASS).
- Не запускалось: `docker compose -f build/docker-compose.yml up -d --build`; рекомендовано выполнить локально после копирования build/.env.example в .env и применения миграций к БД при первом запуске.

**Impact**
- API: не меняется. Данные: postgres — volume postgres_data. Конфигурация только через env. Обратная совместимость: да (добавлены только файлы в build/).

**Risks & Rollback**
- Риск: неверный JWT_SECRET в .env. Откат: удалить/изменить build/docker-compose.yml и Dockerfile’ы, окружение вернуть к ручному запуску.
