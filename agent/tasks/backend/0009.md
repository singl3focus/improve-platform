### Change Summary
**Problem**
- Требовалась формальная проверка backend-контракта относительно реально реализованного frontend MVP.
- По результатам сверки обнаружены критические auth-расхождения: frontend ожидал refresh/logout flow и пару токенов `access+refresh`, а backend предоставлял только `access_token` и не имел маршрутов `/api/v1/auth/refresh`, `/api/v1/auth/logout`.
- После первичного фикса ревью выявило security-дефект: middleware принимал любой валидный JWT (включая `typ=refresh`) как Bearer к защищённым endpoint.

**Root Cause Hypothesis**
- Backend и frontend развивались асинхронно: frontend auth-session слой был построен вокруг refresh-токена, backend auth MVP — вокруг single access token.
- Часть frontend бизнес-экранов (roadmap/tasks/materials/topics/dashboard) работает через локальные Next API mock/in-memory маршруты и не интегрирована с backend.

**Evidence**
- Frontend ожидаемые backend auth paths: `frontend/app/api/auth/_shared.ts` (`login/register/refresh/logout`).
- Frontend auth payload форматы: `frontend/app/api/auth/login/route.ts`, `register/route.ts`, `refresh/route.ts`, `logout/route.ts`.
- До исправления backend не содержал `/api/v1/auth/refresh` и `/api/v1/auth/logout` в `backend/internal/server/server.go`.
- До исправления `backend/internal/auth/model.go` содержал только `access_token` без `refresh_token`.
- До ReMake-исправления `backend/internal/auth/middleware.go` не проверял claim `typ`; проверялись только подпись/`exp`/`sub`.
- Frontend domain routes используют mock/store вместо backend:
  - `frontend/app/api/roadmap/route.ts`
  - `frontend/app/api/tasks/route.ts`
  - `frontend/app/api/materials/route.ts`
  - `frontend/app/api/topics/[topicId]/route.ts`
  - `frontend/app/api/dashboard/*`

**Solution**
- Критический auth-разрыв устранён:
  - `backend/internal/auth/model.go`: `TokenResponse` расширен до `access_token + refresh_token + ttl` (`access_expires_in`, `refresh_expires_in`), добавлена ошибка `ErrInvalidRefreshToken`.
  - `backend/internal/auth/usecase.go`: реализован session issue flow с двумя JWT, добавлены методы `Refresh` и `Logout`, введена проверка refresh-токена.
  - `backend/internal/auth/handler.go`: добавлены HTTP handlers `Refresh` и `Logout` с валидацией payload (`refreshToken` и `refresh_token`), единым error mapping и безопасным ответом.
  - `backend/internal/server/server.go`: зарегистрированы маршруты `POST /api/v1/auth/refresh` и `POST /api/v1/auth/logout`.
- Добавлены contract/integration проверки:
  - `backend/internal/auth/handler_test.go`: тесты на refresh/logout payload, status codes, error schema, ключевые поля ответа.
  - `backend/internal/auth/usecase_test.go`: тесты на выдачу пары токенов и refresh/logout ошибки.
  - `backend/internal/server/frontend_contract_test.go`: route presence/authorization contract для frontend-критичных backend путей.
- ReMake security-исправление:
  - `backend/internal/auth/middleware.go`: добавлена обязательная проверка `claims["typ"] == "access"`; refresh-token теперь не допускается в protected routes.
  - `backend/internal/auth/middleware_test.go`: добавлены тесты `TestMiddleware_RefreshTokenRejected` (401 на refresh-token) и `TestMiddleware_AccessTokenAllowed` (200 на access-token).

**Tests / Verification**
- `go test -v -count=1 ./...` (из `backend/`) — PASS.
- `go build -buildvcs=false ./...` — PASS.
- `go vet ./...` — PASS.
- В contract-тестах подтверждено:
  - backend auth маршруты, ожидаемые frontend (`register/login/refresh/logout`), присутствуют;
  - refresh/logout обрабатывают payload и возвращают корректные статусы/схему ошибок;
  - protected backend routes требуют авторизацию (401 без Bearer);
  - middleware не пускает refresh-token как Bearer (тип токена валидируется).

**Impact**
- API изменения backend:
  - `POST /api/v1/auth/register` и `POST /api/v1/auth/login` теперь отдают пару токенов и TTL (добавочные поля, обратно совместимо для клиентов, читающих только `access_token`).
  - Добавлены `POST /api/v1/auth/refresh` и `POST /api/v1/auth/logout`.
- Безопасность: op-trace остаётся только в серверных логах; клиент получает нормализованные ошибки.
- Обратная совместимость: да (изменения в auth response additive).

**Risks & Rollback**
- Устранённый риск: refresh-token больше нельзя использовать как access-token для защищённых endpoint.
- Риск: `logout` реализован stateless (без blacklist/хранилища refresh-токенов), фактически валидирует токен и возвращает успех.
- Риск: domain-сценарии roadmap/tasks/materials/topics/dashboard остаются mock-only на frontend и ещё не проверяют runtime-контракт backend.
- Откат:
  - удалить новые маршруты refresh/logout из `server.go`;
  - вернуть `auth/model.go`, `auth/usecase.go`, `auth/handler.go`, `auth/middleware.go` и соответствующие тесты к предыдущему состоянию.
