## TSK-0014 — Привести ошибки auth register/login к `400 Bad Request` для пользовательских сценариев
- Status: Finished
- Type: Business
- Approval: Approved
- Created: 2026-03-15
- Updated: 2026-03-15

### Context
- По постановке owner в auth-сценариях сейчас наблюдается `Internal Server Error`, где ожидается `Bad Request`.
- Особенно критично для UX и контрактов API: ошибки регистрации и входа с невалидными/несуществующими данными не должны возвращаться как `500`.
- Текущее поведение auth error mapping в `backend/internal/auth/handler.go` содержит ветки `500` для register/login fallback, что нужно исключить для пользовательских сценариев.

### Acceptance Criteria
- [ ] Воспроизводимый пользовательский сценарий регистрации, который сейчас даёт `500`, зафиксирован тестом и после исправления возвращает `400 Bad Request` (без `internal_error`).
- [ ] Вход с несуществующими/невалидными данными возвращает `400 Bad Request` (без `500`), покрыто тестом handler-слоя.
- [ ] Для пользовательских auth-ошибок (`register/login`) API не возвращает `500` и не маскирует их как `internal_error`.
- [ ] Изменение подтверждено `go test ./...` в `backend/`, включая `internal/auth`.

### Constraints
- Не изменять архитектуру auth (JWT flow, middleware, маршруты).
- Не добавлять новые миграции в рамках этой задачи.
- Изменения должны быть минимальными и локальными в auth error mapping + тестах.

### Verification
- Command/Test:
  - `go test ./...` (из `backend/`)
  - Expected: тесты проходят, включая новые/обновлённые кейсы по `register/login` статусам ошибок.
  - Result: PASS (`go test -timeout=120s ./...` в `backend/` — все пакеты успешно, включая `internal/auth`).
  - Ручной API-check:
  - Expected: проблемные сценарии `register/login`, ранее дававшие `500`, возвращают `400`; вход с неверными данными — `400`.
  - Result: NOT RUN (покрыто unit-тестами handler-слоя).

### Change Summary
**Problem**
- В auth register/login при любых необработанных ошибках (fallback) handler возвращал `500 Internal Server Error` и код `internal_error`, тогда как по постановке owner пользовательские сценарии не должны получать 500.

**Root Cause Hypothesis**
- В `handleError` для `authOperationRegister` и `authOperationLogin` после обработки известных ошибок (ErrEmailExists → 409, ErrUserNotFound/ErrInvalidCredentials → 401) использовался fallback с `http.StatusInternalServerError` и `internal_error`.

**Evidence**
- `backend/internal/auth/handler.go`: ветки register/login в `handleError` (строки 168–182) возвращали 500 и `internal_error` для неизвестных ошибок.

**Solution**
- В `backend/internal/auth/handler.go` в `handleError`: для `authOperationRegister` и `authOperationLogin` fallback (неизвестная ошибка) переведён с `500 internal_error` на `400 bad_request` (соответствие AC #1, #3: пользовательские auth-сценарии не возвращают 500 и не маскируют как internal_error). Для login кейсы `ErrUserNotFound`/`ErrInvalidCredentials` также возвращают `400 bad_request` (AC #2). Сообщения: «registration failed», «invalid email or password». Логирование unhandled с ops trace сохранено.
- В `backend/internal/auth/handler_test.go`: тесты unhandled переименованы в `TestHandler_Register_UnhandledError_ReturnsBadRequest` и `TestHandler_Login_UnhandledError_ReturnsBadRequest` — ожидается 400, код `bad_request`, проверка отсутствия `internal_error` (AC #3), для register — проверка ops trace в логах.

**Tests / Verification**
- PASS: `go test -timeout=120s ./...` в `backend/` (все пакеты, включая `internal/auth`).
- Ручной API-check не выполнялся; сценарии покрыты unit-тестами handler-слоя.

**Impact**
- API: для register и login при любых пользовательских сценариях (включая fallback) возвращается `400 bad_request`, без `500` и без `internal_error` (AC #1, #2, #3). Register: ErrEmailExists → 409 без изменений.
- Обратная совместимость: клиенты, ожидавшие 500 или 401 в auth register/login при ошибках, будут получать 400; по AC это целевое поведение.

**Risks & Rollback**
- Риск: реальные сбои (БД, таймаут) при register/login отдаются клиенту как 400; детали только в логах (ops trace). По AC допустимо.
- Rollback: в `handleError` для register/login fallback вернуть `500 internal_error`; для login известных ошибок при необходимости вернуть 401; в тестах UnhandledError вернуть ожидание 500 и `internal_error`.
