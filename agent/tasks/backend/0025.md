### Change Summary
**Problem**
- Ручная смена статуса roadmap-темы проверяла блокировку только для перехода в `in_progress`.
- Для перехода в `completed` блокировка зависимостями не применялась, хотя бизнес-правило требует запрета ручного перевода blocked-темы и в `in_progress`, и в `completed`.

**Root Cause Hypothesis**
- В `UpdateTopicStatus` проверка `computeBlockage(...)` была ограничена условием `req.Status == "in_progress"`.

**Evidence**
- Код: `backend/internal/roadmap/usecase.go` — блокировка выполнялась только в ветке `if req.Status == "in_progress"`.
- Тесты до изменения покрывали blocked-case только для `in_progress` (`TestUpdateTopicStatus_ToInProgress_Blocked`) и не фиксировали blocked-case для `completed`.

**Solution**
- Расширена серверная проверка blocked-состояния на оба ручных целевых статуса: `in_progress` и `completed`.
- Обновлен unit-тест `TestUpdateTopicStatus_ToCompleted` с учетом новой проверки зависимостей (unblocked сценарий).
- Добавлен unit-тест `TestUpdateTopicStatus_ToCompleted_Blocked`, подтверждающий возврат `ErrTopicBlocked`.
- Добавлен handler-тест `TestHandler_UpdateTopicStatus_BlockedOnCompleted`, подтверждающий нормализацию ошибки в `409 topic_blocked` для frontend/BFF.

**Tests / Verification**

**Impact**
- API: поведение `PATCH /api/v1/roadmap/topics/{topicID}/status` стало строго соответствовать бизнес-правилу для blocked-тем.
- Данные: изменений схемы/миграций нет.
- Безопасность: без изменений.
- Обратная совместимость: да, кроме ожидаемого ужесточения валидации недопустимого ручного перехода blocked-темы в `completed`.

**Risks & Rollback**
- Риск: клиенты, которые ранее полагались на возможность вручную завершать blocked-тему, теперь получают `409 topic_blocked`.
- Rollback: вернуть условие проверки blocked к прежнему виду (только для `in_progress`) в `backend/internal/roadmap/usecase.go` и удалить добавленные тесты.

**Suggested Commit Message**
