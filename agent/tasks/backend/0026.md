## TSK-0026 — Roadmap directional create: сохранить семантику left/right/below после reload
- Status: Finished
- Type: Business
- Approval: Approved
- Created: 2026-03-21
- Updated: 2026-03-21 (Reviewer)

### Context
- На roadmap пользовательский сценарий `Создать ниже` сейчас визуально деградирует после reload: новая карточка оказывается справа вместо нижней ветки дерева.
- До исправления был проверенный конфликт контракта между backend и frontend:
  - в `backend/internal/roadmap/repo.go` логика `below` возвращала `maxPosition + 1`;
  - в `frontend/src/lib/roadmap-layout.ts` направление `below` определяется через равенство `child.position === parent.position`, а `right` через `child.position > parent.position`.
- Пока directional create и layout читали разные семантики позиции, дерево нельзя было считать стабильным после повторной загрузки данных.

### Acceptance Criteria
- [x] `POST /api/v1/roadmap/topics` с `direction=left|right|below` сохраняет такую позиционную семантику, при которой после `GET /api/v1/roadmap` frontend-layout воспроизводит то же направление относительно родительской темы.
- [x] Для сценария `below` после reload дочерняя тема больше не классифицируется как `right` в текущем roadmap-layout контракте.
- [x] Изменение не ломает существующий directional create для `left` и `right`, а также автонормализацию зависимости между parent/created topic.
- [x] Добавлены/обновлены backend-тесты на directional positioning contract и граничные кейсы.

### Constraints
- Не ломать инварианты roadmap DAG и существующий dependency flow.
- Не менять API shape без необходимости; если без этого нельзя, новое поведение должно быть явно задокументировано в задаче и согласовано через Notes.

### Change Summary
**Problem**
- Для directional create backend сохранял `below` как `maxPosition + 1`, из-за чего после `GET /api/v1/roadmap` frontend-layout классифицировал такую тему как `right`, а не `below`.
- Поведение после reload расходилось с текущим layout-контрактом для `below`.

**Root Cause Hypothesis**
- Репозиторный insert-plan для `below` исходил из уникализации `position`, а frontend layout считает ветку `below` только при `child.position === parent.position`.
- Один и тот же directional intent (`below`) интерпретировался разными правилами на backend и frontend.

**Evidence**
- `frontend/src/lib/roadmap-layout.ts:50`-`:56`: `classifyDirection(...)` определяет `left` при `child.position < parent.position`, `right` при `child.position > parent.position`, иначе `below`.
- `backend/internal/roadmap/repo.go:147` и `backend/internal/roadmap/repo.go:192`-`:208`: directional create использует `resolveDirectionalInsertPlan(...)`; для `below` `insertPosition` равен `currentPosition`, а `shiftExisting=false`.
- `backend/internal/roadmap/repo.go:238`: выдача roadmap сортирует темы по `ORDER BY position, created_at`, поэтому duplicate `position` для `below` не ломают стабильную загрузку списка.
- `backend/internal/roadmap/repo_test.go:31`-`:39` и `backend/internal/roadmap/repo_test.go:47`-`:55`: кейсы подтверждают, что `below` сохраняет текущую позицию и не зависит от максимальной позиции в наборе.
- `frontend/src/lib/roadmap-layout.test.ts:354`-`:430`: frontend-тест подтверждает, что при `parent.position == child.position` тема после reload остаётся в ветке `below`, а не уезжает в `right`.
- Проверки:
  - `cd backend && go test -count=1 ./...` -> PASS
  - `cd frontend && npm run test` -> sandbox FAIL (`spawn EPERM`), rerun вне sandbox -> PASS (`50 passed, 0 failed`)
  - `cd backend && golangci-lint run ./...` -> FAIL (`context loading failed: no go files to analyze: running go mod tidy may solve the problem`)
  - `cd backend && go vet ./...` -> PASS
  - `cd backend && go build ./...` -> FAIL (`error obtaining VCS status: exit status 128; Use -buildvcs=false to disable VCS stamping.`)
  - `cd backend && go build -buildvcs=false ./...` -> PASS

**Solution**
- В `backend/internal/roadmap/repo.go` заменён простой расчёт insert-position на `resolveDirectionalInsertPlan(...)`, который возвращает и позицию, и необходимость сдвига существующих тем.
- Для `left` и `right` сохранён прежний collision-safe сдвиг существующих позиций.
- Для `below` новая тема теперь создаётся с `position == currentPosition` и без сдвига существующих тем, чтобы сохранить layout-семантику `below` после reload.
- Обновлены unit-тесты `backend/internal/roadmap/repo_test.go` на directional positioning contract, включая граничный кейс, где `maxPosition` больше `currentPosition`.
- Для ReMake код не менялся: обновлена только документация задачи, чтобы `Verification` и `Evidence` отражали реально воспроизведённые команды и их фактические результаты.

**Impact**
- API shape не менялся; обратная совместимость по контракту запроса/ответа: да.
- Для новых `below`-созданий backend теперь допускает duplicate `position` намеренно, потому что это требуется текущему frontend layout-контракту.
- `left` и `right` сохраняют прежнюю collision-safe семантику со сдвигом позиций.
- Existing roadmap data, созданные старой логикой `below = maxPosition + 1`, автоматически не мигрируются и сохраняют прежнюю позиционную семантику до ручного редактирования/пересоздания.

**Risks & Rollback**
- Risks:
  - Для `below` теперь возможны одинаковые `position` у связанных тем; текущий frontend layout и существующие тесты duplicate positions это поддерживают, но любая логика, которая предполагала уникальность `position`, должна опираться на сортировку с tie-breaker, а не на уникальность.
  - Старые данные с уже сохранённым `below` через `maxPosition + 1` не будут автоматически исправлены этим change-set.
  - `golangci-lint run ./...` в текущем backend-модуле остаётся невоспроизводимым и требует отдельной настройки/уточнения команды; это не новый регресс от фикса directional create, но ограничивает формальную lint-верификацию этой задачи.
- Rollback:
  - вернуть прежнюю логику directional insert-plan в `backend/internal/roadmap/repo.go` и связанные unit-тесты в `backend/internal/roadmap/repo_test.go`.
