- Status: Finished

### Context
- Owner запросил расширить доменную модель материалов: добавить тип материала и единицу измерения, а прогресс считать не как ручной процент, а по выполненной мере.
- Отдельно подтверждены правила единиц:
  - `book`, `article` -> `pages`
  - `course` -> `lessons`
  - `video` -> `hours`
- Для данного проекта допустимо изменить существующую миграцию `000001_initial_schema.up.sql` без добавления новых миграций.

### Acceptance Criteria
- [x] Модель `materials` в backend хранит: тип материала, единицу измерения, полную меру и выполненную меру.
- [x] Единица измерения определяется типом и не принимается как произвольное значение от клиента.
- [x] Прогресс материала в API рассчитывается из выполненной меры и полной меры (с валидной обработкой `0` и граничных значений).

- [x] Обновлена существующая миграция `backend/migrations/000001_initial_schema.up.sql` под новую модель материалов (без создания новых migration-файлов).
- [x] Unit/handler тесты модуля материалов покрывают новые поля, валидации и расчет прогресса.

### Constraints

- Не ломать user-scope ограничения (`topic_id`, `user_id`) и существующие права доступа.

### Change Summary
**Problem**
- Материалы хранили только ручной `progress` (0..100) без доменных полей типа/единицы/меры, из-за чего нельзя было валидно считать прогресс по фактическому выполнению.

**Root Cause Hypothesis**
- Первичная реализация CRUD материалов использовала упрощённую схему (`progress` как число от клиента) и не фиксировала связь `type -> unit`.

**Evidence**
- До изменения таблица `materials` в `backend/migrations/000001_initial_schema.up.sql` содержала колонку `progress` и не содержала `type/unit/total_amount/completed_amount`.
- До изменения `backend/internal/material/model.go` и `backend/internal/material/usecase.go` использовали `Progress` во входных DTO и валидацию диапазона `0..100`.
- ReMake-проверка runtime до фикса: `docker compose -f build/docker-compose.yml up -d --build` на существующем volume приводил к ошибке применения схемы (`type "topic_status" already exists`) и runtime-рассинхрону схемы.
- ReMake-проверка runtime после фикса: `docker compose -f build/docker-compose.yml down; docker compose -f build/docker-compose.yml up -d --build` -> `migrations_reset Exited (0)`, `migrations Exited (0)`, backend стартует без ошибок схемы.
- ReMake-проверка логов: `docker compose -f build/docker-compose.yml logs migrations_reset --tail=120` содержит `DROP SCHEMA`/`CREATE SCHEMA`, `docker compose -f build/docker-compose.yml logs migrations --tail=120` содержит `1/u initial_schema`, `docker compose -f build/docker-compose.yml logs backend --tail=200` не содержит `column "type" does not exist`.
- ReMake-проверка BFF: при поднятом frontend runtime на `127.0.0.1:3025` команда `npm --prefix frontend run verify:mvp` проходит PASS, включая materials-сценарии `GET/POST/PATCH/DELETE`.
- Проверка после изменения: `go test ./internal/material ./internal/server` -> PASS.
- Регрессионная проверка: `go test ./...` -> PASS.

**Solution**
- Обновлена доменная модель материалов (`backend/internal/material/model.go`):
  - добавлены `type`, `unit`, `total_amount`, `completed_amount`;
  - добавлено вычисление `progress` из `completed_amount/total_amount` с обработкой `total_amount == 0`;
  - добавлены константы типов (`book|article|course|video`) и функция маппинга `type -> unit`.
- Обновлены usecase/repo/handler (`backend/internal/material/usecase.go`, `repo.go`, `handler.go`):
  - входные `CreateRequest`/`UpdateRequest` переведены на новые поля;
  - `unit` больше не принимается от клиента, а вычисляется по типу на backend;
  - добавлены валидации `type` и мер (`total_amount >= 0`, `completed_amount >= 0`, `completed_amount <= total_amount`);
  - SQL CRUD переведён на новые колонки.
- Обновлена миграция `backend/migrations/000001_initial_schema.up.sql`:
  - добавлен enum `material_type`;
  - таблица `materials` переведена на `type/unit/total_amount/completed_amount`;
  - добавлены CHECK constraints для единицы по типу и корректности мер.
- Обновлён `backend/migrations/000001_initial_schema.down.sql`: добавлен `DROP TYPE IF EXISTS material_type`.
- Обновлены unit/handler тесты `backend/internal/material/usecase_test.go` и `backend/internal/material/handler_test.go` под новый контракт и валидации.
- ReMake: обновлён runtime-путь запуска в `build/docker-compose.yml` без добавления новых migration-файлов:
  - добавлен сервис `migrations_reset` (Postgres + `psql`) с обязательным `DROP SCHEMA IF EXISTS public CASCADE; CREATE SCHEMA public;`;
  - `migrations` теперь запускается только после успешного завершения `migrations_reset`.
- ReMake: обновлена документация `build/README.md`:
  - добавлено явное предупреждение о destructive reset локальной БД;
  - добавлены шаги проверки логов `migrations_reset` и ожидаемые сигналы успешного reset.

**Impact**
- API: изменён контракт `POST/GET/PUT /api/v1/materials*` на поля `type`, `unit`, `total_amount`, `completed_amount`, вычисляемый `progress`.
- Data model: изменена initial schema таблицы `materials`, добавлен enum `material_type`.
- Runtime: локальный docker-compose теперь принудительно сбрасывает схему БД перед миграциями.
- Безопасность/доступ: user-scope (`topic_id`, `user_id`) и права доступа не изменялись.
- Обратная совместимость: нет, изменение контракта материалов является breaking для клиентов со старым payload.

**Risks & Rollback**
- Риск: клиенты, отправляющие старое поле `progress` и не отправляющие новые поля, получат ошибки валидации.
- Риск: локальный compose-reset удаляет все данные в PostgreSQL volume при каждом старте.
- Rollback plan: убрать сервис `migrations_reset` и зависимость `migrations -> migrations_reset` из `build/docker-compose.yml`, затем поднять окружение заново.
- Rollback plan (для контракта материалов): вернуть `materials`-контракт и схему к варианту с `progress` (файлы `backend/internal/material/*` и `backend/migrations/000001_initial_schema.up.sql`).
