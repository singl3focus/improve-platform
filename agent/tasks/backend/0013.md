## TSK-0013 — Добавить обязательный `full_name` в backend модель пользователя и auth API
- Status: Finished
- Type: Business
- Approval: Approved
- Created: 2026-03-15
- Updated: 2026-03-15

### Context
- По требованию owner поле `full_name` должно быть обязательным при регистрации.
- Сейчас backend хранит и принимает только `email/password`; `full_name` отсутствует в схеме, моделях и auth handler/usecase.
- Для поддержки приветствия на dashboard и профиля пользователя поле должно возвращаться из API (`/api/v1/me`) и участвовать в auth-flow.
- По уточнению owner, база сейчас не используется в production/dev истории, поэтому допустимо изменить существующую initial migration вместо добавления новой.

### Acceptance Criteria
- [x] Обновлена существующая initial migration (`backend/migrations/000001_initial_schema.up.sql`): `users.full_name` присутствует в схеме и является обязательным полем (не `NULL`).
- [x] `POST /api/v1/auth/register` принимает `full_name`, валидирует непустое значение и создает пользователя с этим полем.
- [x] `GET /api/v1/me` возвращает `full_name` в payload пользователя.
- [x] Обновлены backend модели/repo/usecase/handler и тесты (`auth` module) под новое поле.
- [x] Валидационные ошибки по `full_name` возвращаются в предсказуемом формате текущего API error contract.

### Constraints
- Не менять текущую auth-механику токенов и middleware.
- Не ослаблять существующую валидацию email/password.
- Не добавлять новую миграцию для `full_name`; изменить текущую initial migration.
- Сохранить изоляцию данных по пользователям.

### Change Summary
**Problem**
- В auth-потоке отсутствовало поле `full_name`: его не было в initial migration, доменной модели пользователя, регистрации и ответе `/api/v1/me`.

**Root Cause Hypothesis**
- Исходная версия auth-модуля проектировалась только под `email/password`, поэтому `full_name` не был добавлен в SQL-схему и не прокинут через handler/usecase/repo.

**Evidence**
- Схема пользователя обновлена в `backend/migrations/000001_initial_schema.up.sql` (`users.full_name TEXT NOT NULL`).
- Контракт модели/сервиса обновлен в `backend/internal/auth/model.go` (поле `FullName`, сигнатуры `Create` и `Register`).
- SQL-операции создания/чтения пользователя обновлены в `backend/internal/auth/repo.go` (insert/select/returning с `full_name`).
- Проверка тестами: `go test ./...` PASS (включая `internal/auth`).

**Solution**
- В initial migration добавлено обязательное поле `full_name` в таблицу `users`.
- В `auth` модели добавлено поле `FullName` и расширены интерфейсы `Repository`/`Service`.
- В `UseCase.Register` добавлен параметр `fullName` и передача его в repo; `GetCurrentUser` теперь возвращает `full_name`.
- В `Handler.Register` добавлен `full_name` в request payload, валидация непустого значения (`TrimSpace` + `validation_error`) и передача в сервис.
- Обновлены тесты `backend/internal/auth/usecase_test.go` и `backend/internal/auth/handler_test.go`:
  - регистрация с `full_name`;
  - валидация при отсутствии `full_name` (400 + `validation_error`);
  - `/api/v1/me` содержит `full_name`.

**Tests / Verification**
- PASS: `$env:GOCACHE=...; $env:GOTMPDIR=...; go test ./...` в `backend/`.
- NOT RUN: `docker compose -f build/docker-compose.yml up -d --build`.
- NOT RUN: ручной API-check через HTTP-клиент.

**Impact**
- API contract auth расширен обязательным `full_name` в `POST /api/v1/auth/register`.
- Payload `GET /api/v1/me` теперь содержит `full_name`.
- Изменена initial migration (без добавления новой миграции), что соответствует ограничению задачи.
- Обратная совместимость: для клиентов регистрации требуется передавать `full_name`.

**Risks & Rollback**
- Риск: старые клиенты, не отправляющие `full_name`, будут получать `400 validation_error`.
- Риск: изменение initial migration влияет на fresh-развёртывания схемы.
- Rollback: удалить `full_name` из `backend/migrations/000001_initial_schema.up.sql` и откатить изменения в `backend/internal/auth/{model.go,repo.go,usecase.go,handler.go}` и соответствующих тестах.
