### Context
- По доменным требованиям история изменений обязательна и должна включать 2 слоя: технический и бизнес.
- Нужна прослеживаемость изменений статусов, прогресса, дедлайнов и переносов тем.

### Acceptance Criteria
- [x] Реализована запись технической истории (кто/когда/что изменил) для ключевых сущностей.

- [x] Доступен API/механизм чтения истории по сущности с ограничением по пользователю.
- [x] История сохраняется при физическом удалении основных сущностей в рамках согласованной модели.

### Change Summary

**Problem**
- В backend-сервисе отсутствовала запись истории изменений. Без прослеживаемости технических и бизнес-событий невозможно обеспечить аудит действий, отслеживать прогресс и изменения статусов ключевых сущностей.

**Root Cause Hypothesis**
- Не применимо (greenfield-функциональность; таблица `history_events` уже создана в миграции TSK-0002).

**Evidence**
- До начала работы в `internal/` не было модуля `history/`, маршрутизация не включала history-эндпоинты, ни один модуль не записывал события в таблицу `history_events`.

**Solution**
Создан доменный модуль `internal/history/` (4 файла + 2 файла тестов) и интегрирована запись событий в существующие модули:

Структура `internal/history/`:
- `model.go` — доменная модель `Event`, `EventRecord`, DTO `EventResponse`, интерфейсы `EventRecorder` (1 метод), `Repository` (3 метода), `Service` (3 метода, включает `EventRecorder`).
- `repo.go` — `Repo` на `*pgxpool.Pool`: `Insert` (запись события с JSON-сериализацией payload), `ListByEntity` (история по entity_type+entity_id+user_id, ORDER BY created_at DESC), `ListByUser` (история по user_id с опциональной фильтрацией entity_type, пагинация LIMIT/OFFSET).
- `usecase.go` — `UseCase`: `Record` (запись события), `GetEntityHistory` (чтение по сущности), `GetUserHistory` (чтение с фильтрацией и пагинацией, limit clamped 1–200, default 50).
- `handler.go` — `Handler` с HTTP-обработчиком `GetHistory()`: единый endpoint с query-параметрами `entity_type`, `entity_id`, `limit`, `offset`.

Эндпоинт (1, protected JWT middleware):
- `GET /api/v1/history` — чтение истории. Если указаны `entity_type` и `entity_id` — возвращает историю конкретной сущности. Если указан только `entity_type` — фильтр по типу. Без параметров — вся история пользователя. Поддерживает `limit` и `offset` для пагинации.

Интеграция записи событий (паттерн `WithRecorder`):
- В `UseCase` каждого модуля (roadmap, task, material) добавлено поле `recorder history.EventRecorder` и методы `WithRecorder(r) *UseCase`, `record(ctx, ev)`. Recorder опционален (nil-safe); если nil — события не записываются. Ошибки записи логируются через `slog.Error`, но не прерывают основную операцию.
- Конструкторы `NewUseCase` не изменены → существующие тесты работают без правок.

Записываемые события:

Технические (`event_type = "technical"`):
- `entity.created` — при создании roadmap, stage, topic, task, material. Payload: `{"title":"..."}` и другие ключевые поля.
- `entity.updated` — при обновлении roadmap, stage, topic, task, material. Payload: `{"title":"..."}` или `{"field":"status","old_value":"...","new_value":"..."}`.
- `entity.deleted` — при удалении stage, topic, task, material. Payload: `{}`.

Бизнес (`event_type = "business"`):
- `topic.status_changed` — при изменении статуса темы (через roadmap.UpdateTopicStatus или task auto-complete). Payload: `{"old_status":"...","new_status":"..."}` (+ `"trigger":"auto_complete"` при автозавершении).
- `topic.progress_changed` — при изменении прогресса темы из-за выполнения/отмены checklist items. Payload: `{"total":N,"done":M,"percent":P}`.
- `task.completed` — при выполнении checklist item (task → done + topic_id != nil). Payload: `{"task_title":"...","topic_id":"..."}`.
- `topic.stage_changed` — при переносе темы между этапами. Payload: `{"old_stage_id":"...","new_stage_id":"..."}`.
- `task.deadline_changed` — при изменении дедлайна задачи. Payload: `{"old_deadline":"YYYY-MM-DD"|null,"new_deadline":"YYYY-MM-DD"|null}`.

Сохранение при удалении:
- `history_events.entity_id` — обычный UUID без FK на таблицы сущностей. При физическом удалении topic/task/material их записи в `history_events` сохраняются. Каскадное удаление только при удалении пользователя (`user_id` FK → users ON DELETE CASCADE).

Изменённые файлы:
- `internal/roadmap/usecase.go` — добавлено поле `recorder`, методы `WithRecorder`/`record`, запись событий в `CreateRoadmap`, `UpdateRoadmap`, `CreateStage`, `UpdateStage`, `DeleteStage`, `CreateTopic`, `UpdateTopic` (+ stage change detection), `UpdateTopicStatus`, `DeleteTopic`. `UpdateRoadmap` и `UpdateTopic` теперь предварительно читают текущее состояние для обнаружения изменений.
- `internal/task/usecase.go` — добавлено поле `recorder`, методы `WithRecorder`/`record`, запись событий в `CreateTask`, `UpdateTask` (+ deadline change detection), `UpdateTaskStatus` (+ task.completed, topic.progress_changed), `DeleteTask`, `tryAutoCompleteTopic` (+ topic.status_changed с trigger=auto_complete). `UpdateTask` теперь предварительно читает текущую задачу для обнаружения изменения дедлайна.
- `internal/material/usecase.go` — добавлено поле `recorder`, методы `WithRecorder`/`record`, запись событий в `CreateMaterial`, `UpdateMaterial`, `DeleteMaterial`.
- `internal/server/server.go` — добавлен импорт `history`, инициализация `histRepo → histUC → histH`, вызов `WithRecorder(histUC)` на `rmUC`, `taskUC`, `matUC`, новый маршрут `GET /api/v1/history`.

**Impact**
- Новые файлы: `internal/history/` (model.go, repo.go, usecase.go, handler.go, usecase_test.go, handler_test.go).
- Изменённые файлы: `internal/roadmap/usecase.go`, `internal/task/usecase.go`, `internal/material/usecase.go`, `internal/server/server.go`.
- API: 1 новый protected endpoint (`GET /api/v1/history`).
- Обратная совместимость: да — конструкторы `NewUseCase` не изменены, существующие маршруты не затронуты. 86 существующих тестов продолжают проходить.

**Risks & Rollback**
- Риск: `UpdateRoadmap` и `UpdateTopic` в roadmap, `UpdateTask` в task теперь выполняют дополнительный SELECT перед UPDATE для обнаружения изменений. Для MVP это приемлемо; при необходимости можно убрать read-before-write если запись истории этих событий не критична.
- Риск: ошибки записи в `history_events` логируются, но не прерывают основную операцию. При недоступности БД для записи истории события будут потеряны, но основная функциональность продолжит работать.
- Риск: `CountByTopic` вызывается дважды при auto-complete (для записи прогресса и для проверки auto-complete). Для MVP однопользовательский сценарий делает это приемлемым.
- Откат: удалить `internal/history/`, откатить изменения в `roadmap/usecase.go`, `task/usecase.go`, `material/usecase.go`, `server/server.go`.
