### Context
- По доменной модели пункт checklist и задача должны использовать максимально общую модель/флоу.
- Прогресс темы считается только автоматически по checklist items.

### Acceptance Criteria
- [x] Реализована единая модель задач для standalone tasks и checklist items темы.
- [x] Прогресс темы вычисляется автоматически по выполнению checklist items без ручного ввода процента.
- [x] При выполнении всех checklist items тема автоматически получает статус `завершено`.
- [x] Просроченность задачи вычисляется динамически по дедлайну и не хранится отдельным статусом.

### Constraints
- Статусы задач: `новая`, `в работе`, `на паузе`, `выполнена`.
- Статусы темы: `не начато`, `в работе`, `на паузе`, `завершено`.
- Нельзя ломать обратную совместимость API, если она уже используется frontend.

### Change Summary

**Problem**
- В backend-сервисе отсутствовал API для управления задачами — ключевой бизнес-сущностью MVP. Без единой модели задач невозможно реализовать checklist тем, автоматический прогресс и автозавершение тем.

**Root Cause Hypothesis**
- Не применимо (greenfield-функциональность; таблица `tasks` и ENUM `task_status` уже созданы в миграции TSK-0002).

**Evidence**
- До начала работы в `internal/` не было модуля `task/`, маршрутизация не включала task-эндпоинты.

**Solution**
Создан доменный модуль `internal/task/` (4 файла + 2 файла тестов):

Структура:
- `model.go` — доменная модель `Task`, request/response DTO (`CreateRequest`, `UpdateRequest`, `UpdateStatusRequest`, `TaskResponse`, `TopicTasksResponse`), sentinel errors (`ErrTaskNotFound`, `ErrInvalidStatus`), интерфейсы `Repository` (7 методов), `TopicStatusUpdater` (2 метода), `Service` (7 методов).
- `repo.go` — `Repo` на `*pgxpool.Pool`: полный CRUD для tasks. `Create` с опциональным `topic_id` (NULL = standalone task, non-NULL = checklist item). `ListByUser` с опциональной фильтрацией по `topic_id`. `CountByTopic` с `COUNT(*) FILTER (WHERE status = 'done')` для вычисления прогресса. Фильтрация всех операций по `user_id`.
- `usecase.go` — `UseCase`: бизнес-логика включая:
  - **Status transitions**: валидная матрица `{new→in_progress, in_progress→paused|done, paused→in_progress|new, done→in_progress}`.
  - **Auto-complete topic**: при переводе задачи в `done`, если задача привязана к теме (`topic_id != nil`), вызывается `tryAutoCompleteTopic` — подсчёт total/done по теме; если all done и topic status = `in_progress`, тема автоматически переводится в `completed` с установкой `completed_date`.
  - **Dynamic overdue**: `is_overdue` вычисляется в `buildTaskResponse` по формуле `deadline != nil && status != done && today > deadline`. Не хранится в БД.
  - **Topic progress**: `GetTopicTasks` возвращает `total`, `done`, `percent` (integer division) и список задач темы.
- `handler.go` — `Handler` с 7 HTTP-обработчиками и централизованной функцией `handleError` для маппинга domain errors → HTTP status codes.

Эндпоинты (7, все protected JWT middleware):
- Tasks CRUD: `POST /api/v1/tasks`, `GET /api/v1/tasks` (?topic_id=), `GET /api/v1/tasks/{taskID}`, `PUT /api/v1/tasks/{taskID}`, `DELETE /api/v1/tasks/{taskID}`
- Status: `PATCH /api/v1/tasks/{taskID}/status`
- Topic tasks with progress: `GET /api/v1/roadmap/topics/{topicID}/tasks`

Единая модель:
- Задача с `topic_id = NULL` — standalone task (появляется в общем списке задач пользователя).
- Задача с `topic_id = <UUID>` — checklist item темы (появляется и в общем списке, и в `GET .../topics/{topicID}/tasks`).
- CRUD, статусные переходы и overdue-вычисления идентичны для обоих типов.

Формат ответа `GET /api/v1/roadmap/topics/{topicID}/tasks`:
```json
{
  "topic_id": "...",
  "total": 3,
  "done": 2,
  "percent": 66,
  "tasks": [{ "id": "...", "status": "done", "is_overdue": false, ... }, ...]
}
```

Изменённые файлы:
- `internal/server/server.go` — добавлена инициализация task-модуля (`repo → usecase → handler`) через `topicStatusAdapter` (адаптер `roadmap.Repository` → `task.TopicStatusUpdater`) и 7 маршрутов в protected-группе Chi.

**Impact**
- Новые файлы: `internal/task/` (model.go, repo.go, usecase.go, handler.go, usecase_test.go, handler_test.go).
- Изменённые файлы: `internal/server/server.go`.
- API: 7 новых protected endpoint.
- Обратная совместимость: новые маршруты добавлены в существующую protected-группу; существующие маршруты (roadmap, auth) не затронуты.

**Risks & Rollback**
- Риск: auto-complete topic вызывает `roadmap.Repository.UpdateTopicStatus` напрямую через адаптер, обходя бизнес-логику roadmap UseCase (blockage check). Для MVP это приемлемо: auto-complete переводит только `in_progress → completed`, что не требует blockage check (блокировка проверяется при переходе В `in_progress`, а не ИЗ него).
- Риск: `CountByTopic` и `UpdateStatus` не выполняются в одной транзакции. Race condition при одновременных запросах теоретически возможен. Для MVP однопользовательский сценарий делает это приемлемым.
- Откат: удалить `internal/task/`, откатить `server.go`.
