## TSK-0024 — API направленного создания тем roadmap (left/right/below) с защитой от коллизий позиций
- Status: Finished
- Type: Business
- Approval: Approved
- Created: 2026-03-20
- Updated: 2026-03-20

### Context
- Владелец подтвердил целевую модель графа: от текущей темы пользователь должен создавать новую тему справа, слева или ниже, с формированием стрелочной связи от текущей темы к созданной.
- Текущее backend-API не выражает направленное создание темы как отдельную бизнес-операцию и не гарантирует удобный контракт для tree-like сценария UI.

### Acceptance Criteria
- [x] В backend добавлен/обновлен явный API-контракт направленного создания темы относительно текущей (`right`, `left`, `below`) без двусмысленности параметров.
- [x] Для каждого направления backend детерминированно вычисляет `position` новой темы так, чтобы исключать коллизию позиций в результирующем наборе тем.
- [x] При создании темы в любом из направлений автоматически создается корректная dependency-связь от текущей темы к новой теме.
- [x] Валидации (self-link, duplicate dependency, cycle) сохраняются и покрыты тестами usecase/handler/repo на новые сценарии.
- [x] Существующий child-create поток заменяется на семантику `below` в API и больше не является отдельным специальным режимом.

### Constraints
- Не нарушать DAG-инварианты roadmap.
- Не ломать user-scope ограничения доступа к данным.
- Не допускать частично успешных операций: создание темы и создание dependency должны быть атомарны.
- Не использовать git-команды.

### Change Summary
**Problem**
- Backend topic-create контракт не выражал направленное создание относительно текущей темы (`left/right/below`), а child-flow делался как отдельная последовательность create + dependency.
- В таком потоке не было атомарности между созданием темы и автосвязью, и не было встроенного collision-safe поведения для позиции в рамках направленного create.

**Root Cause Hypothesis**
- `POST /api/v1/roadmap/topics` принимал только `position` и не содержал явной модели направленного create.
- Репозиторный слой не имел транзакционной операции «создать тему + сразу создать dependency», поэтому шаги могли быть частично успешными.

**Evidence**
- Код до изменений:
  - `backend/internal/roadmap/model.go`: `CreateTopicRequest` содержал только `title/description/position`.
  - `backend/internal/roadmap/usecase.go`: `CreateTopic` вызывал только `repo.CreateTopic(...)`.
  - `backend/internal/roadmap/repo.go`: отсутствовал атомарный directional create-метод.
- Проверки после изменений:
  - `cd backend && go test ./internal/roadmap ./internal/server` -> PASS
  - `cd backend && go test ./...` -> PASS
  - `cd backend && golangci-lint run ./...` -> PASS (`0 issues`)
  - `cd backend && go vet ./...` -> PASS
  - `cd backend && go build ./...` -> PASS
  - `cd frontend && npm run verify:mvp` -> PASS (при запущенном `next start` на `127.0.0.1:3025`)

**Solution**
- Расширен контракт create-topic:
  - добавлены `direction` (`left|right|below`) и `relative_to_topic_id` в `CreateTopicRequest`.
  - добавлен enum-type `TopicCreateDirection` + валидация `IsValid()`.
- Добавлена явная directional-ветка в usecase:
  - `CreateTopic` теперь поддерживает directional create через новый репозиторный метод.
  - некорректная directional-конфигурация возвращает `ErrInvalidDirection`.
- Реализована атомарная операция в repo:
  - новый `CreateTopicDirectional(...)` выполняется в транзакции.
  - внутри транзакции блокируются темы пользователя (`FOR UPDATE`), вычисляется позиция по направлению, выполняется сдвиг `position` для исключения коллизий, создаётся тема и сразу создаётся dependency `current_topic -> new_topic`.
- Обновлён handler:
  - добавлена валидация directional-полей (`relative_to_topic_id`, `direction`) в `CreateTopic`.
  - добавлен error-mapping для `ErrInvalidDirection`.
- Добавлены тесты новых сценариев:
  - `usecase`: directional success + invalid direction.
  - `handler`: directional success + invalid payload scenarios.
  - `repo`: deterministic position resolver + direction validation helper.

**Impact**
- API/контракт:
  - `POST /api/v1/roadmap/topics` теперь поддерживает направленный режим через `direction` + `relative_to_topic_id`.
  - Базовый create по `position` сохранён (обратная совместимость: да).
- Данные:
  - directional create теперь гарантирует уникализацию `position` через транзакционный сдвиг.
  - dependency создаётся в той же транзакции, что и тема.
- Безопасность:
  - user-scope сохраняется (все SQL-фильтры по `user_id`).

**Risks & Rollback**
- Risks:
  - directional create теперь блокирует строки тем пользователя (`FOR UPDATE`) на время транзакции; при высокой конкуренции возможен рост latency.
- Rollback:
  - откатить изменения в `backend/internal/roadmap/model.go`, `backend/internal/roadmap/usecase.go`, `backend/internal/roadmap/repo.go`, `backend/internal/roadmap/handler.go` и новые тесты, вернув прежний non-directional flow.
